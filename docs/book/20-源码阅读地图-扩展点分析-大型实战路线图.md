## 先建立整体认知

Microsoft Agent Framework for .NET 的源码阅读，不建议从最复杂的端到端示例开始，也不建议只看抽象接口。更稳妥的顺序是：

1. 先看 `dotnet/samples/01-get-started`，建立最小运行链路。
2. 再看 `dotnet/src/Microsoft.Agents.AI.Abstractions` 和 `dotnet/src/Microsoft.Agents.AI`，理解核心抽象与管道。
3. 然后看 Provider 与集成层，例如 `Microsoft.Agents.AI.OpenAI`、`Microsoft.Agents.AI.Anthropic`、`Microsoft.Agents.AI.Foundry`、`Microsoft.Agents.AI.AzureAI.Persistent`。
4. 接着看 `Microsoft.Agents.AI.Workflows`、`Microsoft.Agents.AI.DurableTask` 和对应 samples，理解长流程与可靠编排。
5. 最后看 Hosting、A2A、AG-UI、DevUI、Purview、CosmosNoSql、Mem0、MCP、Declarative 等扩展层。

本章的核心认知是：**大型 Agent 应用不是一个“大 Agent”，而是一组可替换的 Agent 运行时、模型 Provider、工具层、知识层、流程层、状态层、入口层、治理层的组合**。

如果缺少这一层源码地图，你会遇到三类问题：

- sample 能跑，但不知道生产项目应该依赖哪些包、改哪些扩展点。
- 出问题时只会调 Prompt，不知道是会话、Tool、Provider、Workflow、Hosting 还是权限边界出了问题。
- 系统变复杂后无法写测试、做回归、接入观测和治理。

## 核心概念拆解

### 1. 源码阅读地图是什么

源码阅读地图不是目录清单，而是“从调用入口到扩展点”的路径。建议按以下主线阅读：

```text
sample Program.cs
  -> Agent 创建 / Provider 创建
  -> AIAgent.RunAsync 或 RunStreamingAsync
  -> AgentSession / AgentRunOptions / ChatMessage
  -> Tool / AIContext / ChatHistory
  -> Provider 适配层
  -> Workflow 或 Hosting 包装
  -> tests 中的对应行为验证
```

### 2. 扩展点分析是什么

扩展点不是“能改的代码”，而是框架有意开放的替换边界。典型扩展点包括：

- 替换模型 Provider：例如从 OpenAI 切到 Anthropic、Foundry 或 Azure AI Persistent。
- 包装 Agent 行为：通过 `AIAgentBuilder`、`DelegatingAIAgent` 增加日志、权限检查、成本统计、请求改写、失败重试。
- 扩展会话与上下文：通过 `AgentSession`、`ChatHistoryProvider`、`AIContextProvider` 连接业务状态、历史存储和 RAG 上下文。
- 扩展 Tool：函数工具、MCP 工具、Agent-as-tool、Workflow-as-agent。
- 扩展 Workflow：节点、条件边、并发、子工作流、Checkpoint、HITL。
- 扩展入口协议：Hosting、A2A、AG-UI、Azure Functions、WebChat。

### 3. 大型实战路线图是什么

大型实战路线图是把 sample 能力分阶段拼成系统：

1. 最小可用：单 Agent + Prompt + 一个只读 Tool。
2. 可用原型：多轮会话 + 工单查询/创建工具 + 基础 Workflow。
3. 复杂系统：RAG + 多 Agent 分工 + MCP/外部系统 + 审批节点。
4. 生产服务：Hosting + DurableTask + 可观测性 + 测试评测 + 安全成本治理。
5. 平台演进：插件化 Provider、统一 Tool Registry、统一审计、租户隔离、灰度发布、ADR 与回归基线。

### 4. 常见误解是什么

- 误解一：只要 Prompt 足够好就能做大型应用。实际大型应用更依赖边界、状态、工具权限、恢复、评测和治理。
- 误解二：sample 就是生产架构。sample 是最小学习入口，生产项目需要补身份认证、审计、限流、监控、回归和运维策略。
- 误解三：Workflow 一定替代 Agent。实际二者是互补关系：Agent 负责语言理解与决策，Workflow 负责可控流程与可靠执行。
- 误解四：扩展点就是改框架源码。优先使用公开抽象和组合方式；只有明确需要贡献框架能力时才修改 `dotnet/src`。

## 结合源码理解执行过程

### 1. 入口在哪里

推荐从 sample 入口读起：

- 最小入口：`dotnet/samples/01-get-started/01_hello_agent/Program.cs`。
- Tool 入口：`dotnet/samples/01-get-started/02_add_tools/Program.cs`。
- Workflow 入口：`dotnet/samples/01-get-started/05_first_workflow/Program.cs`。
- Hosting 入口：`dotnet/samples/01-get-started/06_host_your_agent` 与 `dotnet/samples/04-hosting`。
- 端到端入口：`dotnet/samples/05-end-to-end/M365Agent/Program.cs`、`AFAgentApplication.cs`、`Agents/WeatherForecastAgent.cs`。

### 2. 调用是如何继续流转的

一个典型 Agent 请求可以按下面理解：

```text
用户请求
  -> sample/host 接收输入
  -> 构造 ChatMessage / AgentRunOptions / AgentSession
  -> 调用 AIAgent.RunAsync 或 RunStreamingAsync
  -> Agent 管道执行装饰逻辑，例如日志、审批、上下文注入
  -> Provider 把请求转换为模型服务调用
  -> 模型可能触发 Tool / MCP / Agent-as-tool
  -> 会话历史、上下文、状态被更新
  -> 返回 AgentResponse 或 AgentResponseUpdate
  -> host 将结果返回 UI、API、A2A 或 AG-UI 客户端
```

### 3. 关键对象分别负责什么

- `AIAgent`：统一运行抽象，调用者不应直接绑定某个模型 SDK。
- `AIAgentBuilder`：把一个基础 Agent 包装成带中间件/管道能力的 Agent。
- `AgentSession`：承载会话与线程语义，是多轮对话和状态恢复的关键入口。
- `AgentRunOptions`：承载一次运行的选项与附加属性，适合传递业务上下文、控制参数和追踪信息。
- `AgentResponse` / `AgentResponseUpdate`：分别面向一次性响应和流式响应。
- `ChatHistoryProvider`：连接框架历史与外部存储。
- `AIContextProvider`：把额外上下文注入 Agent，例如 RAG 检索结果或业务侧上下文。
- Workflow 相关类型：负责确定节点、边、执行器、检查点与恢复语义。
- Hosting 相关类型：负责把 Agent 暴露给 Web、Azure Functions、A2A、AG-UI 等入口。

### 4. 扩展点在哪里

- 读 `dotnet/src/Microsoft.Agents.AI/AIAgentBuilder.cs`：理解如何用 `Use(...)` 包装 Agent。
- 读 `dotnet/src/Microsoft.Agents.AI/AgentExtensions.cs`：理解 Agent 如何转成函数工具、如何作为组合单元。
- 读 `dotnet/src/Microsoft.Agents.AI.Abstractions/AIAgent.cs`：理解运行抽象边界。
- 读 `dotnet/src/Microsoft.Agents.AI.Abstractions/AgentSession.cs` 与 `ChatHistoryProvider.cs`：理解会话和历史扩展。
- 读 `dotnet/src/Microsoft.Agents.AI.Workflows`：理解 Workflow 核心执行模型。
- 读 `dotnet/src/Microsoft.Agents.AI.DurableTask`：理解可靠长任务和持久化编排。
- 读 `dotnet/src/Microsoft.Agents.AI.Hosting*`：理解服务化入口。
- 读 `dotnet/tests/*UnitTests`：确认某个扩展点的预期行为，而不是只靠猜测。

## 关键代码或配置讲解

### 代码片段 1：`AIAgentBuilder.Use(...)`

- 作用：在基础 Agent 外层增加一个包装层，例如日志、权限检查、上下文注入、重试、成本统计。
- 为什么重要：大型应用不能把这些横切逻辑散落在每个业务 Agent 中，应该通过管道统一治理。
- 改动它会影响什么：会影响所有通过该 builder 构造出的 Agent。生产中应谨慎处理异常传播、取消令牌、流式响应和敏感信息脱敏。

### 代码片段 2：`AgentExtensions.AsAIFunction(...)`

- 作用：把一个 Agent 暴露为可调用函数，让多 Agent 或 Tool 编排变得统一。
- 为什么重要：企业内部知识与工单协同系统中，可以把“知识检索 Agent”“工单分类 Agent”“审批建议 Agent”作为上层 Router 的工具。
- 改动它会影响什么：会影响 Agent-as-tool 的参数、返回、上下文传递和会话隔离策略。

### 代码片段 3：`dotnet/samples/03-workflows/Checkpoint` 系列

- 作用：展示 Workflow 在检查点、恢复、人工介入场景下的执行方式。
- 为什么重要：工单处理、变更审批、运维修复等流程天然需要暂停、恢复、人工确认。
- 改动它会影响什么：会影响任务恢复语义、幂等性、外部系统副作用和审计记录。

### 代码片段 4：`dotnet/samples/05-end-to-end/M365Agent`

- 作用：展示端到端应用如何把 Agent 类、应用入口、认证配置、响应内容类型组合起来。
- 为什么重要：它比 get-started 更接近真实应用边界，可以作为综合案例“企业内部知识与工单协同 Agent 系统”的应用壳参考。
- 改动它会影响什么：会影响认证、manifest、用户入口、业务 Agent 类和部署形态。

## 做一个最小改造实验

### 实验目标

在不修改框架源码的前提下，为一个现有 Agent 增加“源码阅读型扩展”：记录每次运行的输入摘要、会话标识、是否调用工具、响应耗时。

### 改造点

- 选择 `dotnet/samples/01-get-started/02_add_tools` 或 `dotnet/samples/02-agents/Agents/Agent_Step05_Observability`。
- 使用 `AIAgentBuilder` 包装原有 Agent。
- 在 wrapper 中记录：开始时间、`AgentRunOptions` 中的附加属性、响应结果摘要、异常。
- 不记录完整用户隐私内容和密钥。

### 你应该观察什么

- Agent 原本能力不变，但多了统一日志。
- Tool 调用失败时，你能区分“模型没选择工具”“工具参数错误”“外部系统失败”“Provider 调用失败”。
- 流式响应场景下，应确认包装逻辑不会破坏 `RunStreamingAsync` 的枚举行为。

### 可能出现的问题

- 只包装了 `RunAsync`，忘记处理 `RunStreamingAsync`。
- 日志中泄露用户输入、token、连接字符串或外部系统返回的敏感数据。
- 在 wrapper 中吞掉异常，导致上层 Hosting 或 Workflow 无法感知失败。
- 未传递 `CancellationToken`，导致请求取消后后台仍在运行。

## 常见问题与排错

### 问题 1：不知道该先读哪个项目

- 现象：打开 `dotnet/src` 后项目很多，读了半天仍不知道调用链。
- 原因：直接从 Provider 或 Hosting 读起，缺少核心抽象。
- 排查方法：先看 sample 的 `Program.cs`，再跳到 `AIAgent`、`AIAgentBuilder`、`AgentSession`。
- 修复方式：按“sample -> Abstractions -> AI -> Provider -> Workflow -> Hosting -> tests”的顺序阅读。

### 问题 2：sample 运行失败

- 现象：缺少 API key、模型部署名、Azure 资源、M365 权限或认证配置。
- 原因：端到端 sample 往往依赖真实外部服务。
- 排查方法：先读对应 sample 的 README、`appsettings.json.template`、项目文件和 `Program.cs`。
- 修复方式：优先运行 `01-get-started` 中依赖最少的示例；复杂 sample 先阅读源码，再按环境补配置。

### 问题 3：自定义扩展后流式响应异常

- 现象：非流式调用正常，流式调用没有输出或提前结束。
- 原因：包装 Agent 时只处理 `RunAsync`，没有正确转发 `RunStreamingAsync`。
- 排查方法：搜索 `AIAgentBuilder.Use(...)`、`DelegatingAIAgent`，检查是否同时覆盖两类调用。
- 修复方式：对流式响应使用异步枚举透传，并正确处理异常、取消和 finally 逻辑。

### 问题 4：Workflow 恢复后重复调用外部系统

- 现象：工单被创建两次，通知发了两次。
- 原因：长流程恢复时没有为副作用工具设计幂等键。
- 排查方法：查看 Checkpoint 与 Durable sample，确认外部写操作是否有业务唯一键。
- 修复方式：为写操作增加 request id、operation id 或 ticket draft id，并在外部系统层做幂等。

### 问题 5：测试无法覆盖真实模型不确定性

- 现象：本地跑通，CI 或生产表现波动。
- 原因：只依赖人工试跑，没有建立单元测试、集成测试、评测数据集和 mock/stub。
- 排查方法：参考 `dotnet/tests` 下 UnitTests、IntegrationTests、AgentConformance.IntegrationTests 的分层。
- 修复方式：把核心业务规则放到可确定测试中，把模型输出放到评测和回归集中，把 Provider 行为用适配层隔离。

## 企业级落地提醒

- 稳定性：长任务、人工审批、外部系统写操作必须结合 DurableTask、Checkpoint、幂等和重试策略设计。
- 可维护性：不要在一个 Agent 中堆所有能力；按 Router、Specialist、Tool、Workflow、State、Governance 分层。
- 权限边界：Tool 需要按用户身份和租户隔离授权；Agent 的“建议”不能自动升级为“执行权限”。
- 安全风险：Prompt Injection、工具参数注入、RAG 污染、敏感信息泄露都要在 Tool 和上下文层拦截，而不是只靠系统 Prompt。
- 成本控制：在 Agent wrapper 或 Hosting 层记录模型、token、工具调用次数和失败重试次数。
- 可观测性：统一 trace id、user id、session id、workflow instance id、tool call id，避免问题发生后无法串联。
- 可测试性：用 `dotnet/tests` 的分层思路建立业务测试；把不可控模型输出和可控业务规则分开验证。
- 持久化与恢复：会话历史、任务状态、审批状态、外部系统操作状态要有明确归属，不要只保存在内存中。
- 工程化：`dotnet/eng/verify-samples`、`eng/scripts` 和 MSBuild 配置说明 sample 与包验证是工程体系的一部分，企业项目也应建立类似验证脚本。
- 架构决策：参考 `docs/decisions` 的 ADR 风格记录 Provider 选择、存储选择、审批边界、观测字段和成本策略。

## 本章小结

- 本章最重要的结论 1：源码阅读要从 sample 入口和核心抽象开始，再进入 Provider、Workflow、Hosting 与 tests。
- 本章最重要的结论 2：扩展点优先使用公开抽象、builder、provider、session、tool、workflow 和 hosting 组合，不要轻易修改框架源码。
- 本章最重要的结论 3：大型 Agent 应用的关键不只是模型调用，而是状态、工具权限、流程恢复、观测、评测、安全、成本和工程化治理的整体闭环。
