## 先建立整体认知

### 这一章真正要解决的不是“安装软件”

很多人把环境准备理解成：装一个 .NET SDK，配置几个环境变量，然后 `dotnet run` 一下就结束了。

这在普通教程里可能够用，但对这个仓库不够。因为这个仓库不是一个单一 demo，而是一个包含：

- 多语言实现（.NET / Python）
- 多层源码（抽象层、实现层、托管层、工作流层）
- 多类 samples（入门、Agent、Workflow、Hosting、端到端）
- 多套测试（UnitTests、IntegrationTests）

的大型代码仓库。

所以本章的核心任务其实有两个：

1. **把运行环境准备好**
2. **把仓库阅读地图建立起来**

如果只做第一个，不做第二个，后面你仍然会卡在：

- 不知道应该先看 `01-get-started` 还是 `02-agents`
- 不知道 `src` 和 `samples` 的关系
- 不知道某个 sample 展示的是框架能力，还是业务型 demo
- 不知道看哪个项目更接近后面要写自己的 Agent

### 在 Agent 系统里，这一层属于“开发起步层”

从整个 Agent 系统学习路径看，环境准备与仓库结构不属于框架运行时本身，而属于**开发者认知基础设施**。

它决定了你后面是否能高效完成这些事情：

- 找到正确 sample
- 把 sample 和底层源码对应起来
- 区分“框架核心能力”和“示例工程写法”
- 知道自己现在应该读 API、读 sample，还是读测试

如果缺少这一层，后面每一章都会出现理解偏移：你看到代码，但不知道它在全局里处于哪里。

### `dotnet/` 下最重要的三层

#### `dotnet/src`
这是框架源码区，重点看“框架提供了什么”。

例如当前可以看到的核心项目包括：

- `Microsoft.Agents.AI`
- `Microsoft.Agents.AI.Abstractions`
- `Microsoft.Agents.AI.OpenAI`
- `Microsoft.Agents.AI.Workflows`
- `Microsoft.Agents.AI.Hosting`
- `Microsoft.Agents.AI.DurableTask`
- `Microsoft.Agents.AI.Declarative`

粗略理解可以先这样分：

- `Abstractions`：抽象层、通用契约
- `AI`：Agent 运行核心能力
- `OpenAI` / `Foundry` / `Anthropic` 等：具体模型或提供方适配层
- `Workflows`：图式工作流与编排能力
- `Hosting`：宿主与服务化相关能力
- `DurableTask`：持久化、长任务、可恢复执行相关能力
- `Declarative`：声明式定义能力

#### `dotnet/samples`
这是学习入口区，重点看“怎么用”。

根据 `dotnet/samples/README.md`，其结构是：

- `01-get-started/`：渐进式入门，hello agent → hosting
- `02-agents/`：按概念展开，包含 tools、providers、memory、middleware 等
- `03-workflows/`：工作流、多步骤编排、条件分支、并发、声明式 workflow
- `04-hosting/`：A2A、Durable Agents、Durable Workflows、Azure Functions
- `05-end-to-end/`：更接近真实系统的完整示例

#### `dotnet/tests`
这是验证与补细节的区域，重点看“框架实际保证了什么”。

当你在 sample 里看到某个概念，但想知道其边界条件、错误处理、序列化方式、事件流细节时，测试通常比 sample 更可靠。

对于初学者，本章不建议一开始就钻测试，但你至少要知道：

- sample 教你怎么用
- src 告诉你框架怎么实现
- tests 告诉你实现被如何验证

- 验证本地 .NET SDK 是否满足仓库要求

优先建议：

1. 进入示例目录：


2. 设置环境变量：

## 核心概念拆解

### 1. 环境准备不只是“装 SDK”

在这个仓库里，环境准备至少包含四层：

- **工具链层**：.NET SDK、NuGet 恢复能力、PowerShell / 终端能力
- **认证层**：Azure CLI 登录、OpenAI API Key 或其他 provider 凭据
- **配置层**：环境变量、`appsettings`、sample 自身要求
- **仓库认知层**：知道该从哪一层 sample 开始看

只装了 SDK，不代表环境已经准备完成。

### 2. `global.json` 是什么

`dotnet/global.json` 用来约束仓库期望使用的 SDK 版本。

当前仓库可见配置是：

```json
{
  "sdk": {
    "version": "10.0.200",
    "rollForward": "minor",
    "allowPrerelease": false
  }
}
```

它的意义是：

- 仓库希望你用 .NET 10 SDK
- 可以在小版本内滚动
- 不默认接受预览版 SDK

如果你本机只有更老的大版本 SDK，那么很多 sample 甚至无法 restore/build。

### 3. `Directory.Packages.props` 是什么

这个文件启用了集中包版本管理：

- `ManagePackageVersionsCentrally=true`
- `CentralPackageTransitivePinningEnabled=true`

它说明：

- 包版本不是散落在每个 `.csproj` 各自定义
- 整个 `dotnet/` 目录下大量项目共享一套版本基线
- 当你看 sample 引用包时，不要误以为“没写 Version 就是不完整”，版本可能在中央配置里统一管理

这对后面阅读 sample 很重要，因为你会经常看到 `.csproj` 很干净，但并不代表依赖少。

### 4. `src`、`samples`、`tests` 各自解决什么问题

#### `src`
回答：**框架提供了什么能力、这些能力如何组织**

#### `samples`
回答：**作为开发者，应该怎么用这些能力**

#### `tests`
回答：**这些能力在真实代码里被如何验证、有哪些边界条件**

这三者的关系不是并列文档，而是一条很重要的阅读链：

- 先从 `samples` 找感知
- 再从 `src` 看实现
- 再用 `tests` 补边界和验证方式

### 5. 为什么 `01-get-started` 是最优先入口

`dotnet/samples/README.md` 已经明确给出推荐顺序：

1. `01_hello_agent`
2. `02_add_tools`
3. `03_multi_turn`
4. `04_memory`
5. `05_first_workflow`
6. `06_host_your_agent`

这说明仓库作者已经帮你设计好了认知递进：

- 先有单 Agent
- 再加 Tool
- 再理解多轮会话
- 再理解 Memory
- 再进入 Workflow
- 最后进入 Hosting

这恰好和本书前五章的安排高度一致。

### 6. 常见误解是什么

#### 误解一：我只需要看 sample，不需要看 src
不对。sample 只能告诉你怎么用，不能告诉你设计边界在哪里。

#### 误解二：我只需要看 `dotnet/README.md`
不对。`dotnet/README.md` 只是入口，真正的学习密度在 `dotnet/samples` 和 `dotnet/src`。

#### 误解三：仓库目录很多，必须先全部看完再学
不对。你需要的是有优先级的地图，而不是完整扫描。

#### 误解四：一跑失败就说明框架很复杂
很多时候不是框架复杂，而是：

- SDK 版本不匹配
- 环境变量没设
- 没有 `az login`
- 账号没有资源权限

## 结合源码理解执行过程

这一章不以某个高级能力为中心，但仍然可以借 `01_hello_agent` 建立一条最小执行链路。

### 1. 入口在哪里

- 文件：`dotnet/samples/01-get-started/01_hello_agent/Program.cs`

这是一个顶层程序文件，直接创建 Agent 并发起调用。

### 2. 调用是如何继续流转的

从代码可见，最小链路是：

1. 读取环境变量
2. 创建 `AzureOpenAIClient`
3. 获取 `ChatClient`
4. 调用 `.AsAIAgent(...)` 构造 Agent
5. 调用 `RunAsync(...)`
6. 调用 `RunStreamingAsync(...)`
7. 输出文本结果/流式更新

也就是：

**配置 -> Provider Client -> Agent 包装 -> Agent 运行 -> 输出结果**

### 3. 关键对象分别负责什么

#### `AzureOpenAIClient`
负责连接具体模型服务。

#### `ChatClient`
代表底层聊天式模型调用入口。

#### `AIAgent`
把底层模型能力提升为 Agent 抽象，让你后面可以继续叠加：

- Prompt
- Tool
- Session
- Memory
- Middleware
- Workflow 集成

#### `RunAsync` / `RunStreamingAsync`
分别对应：

- 一次性获取结果
- 流式接收更新

### 4. 扩展点在哪里

从第二章视角，你现在先记住这些最常见扩展方向即可：

- 换 provider：去 `02-agents/AgentProviders`

- 加 memory：去 `01-get-started/04_memory`
- 加 workflow：去 `01-get-started/05_first_workflow`

## 做一个最小改造实验

### 实验目标

验证你已经不只是“能跑”，而是“开始看懂仓库结构和最小修改路径”。

### 改造点

在 `dotnet/samples/01-get-started/01_hello_agent/Program.cs` 中，先不要动框架代码，只改这一行：

- `instructions: "You are good at telling jokes."`

把它改成更明确的角色，例如：

- 企业内部知识助理
- 工单分诊助手
- IT 支持问答机器人

再重新运行。

### 你应该观察什么

- 输出风格是否变化
- 同样的用户输入，回答内容是否更偏向新角色
- 你能否明确区分：角色变化来自 sample 改动，而不是框架内部变了

### 这和综合案例的关系

这是主案例“企业内部知识与工单协同 Agent 系统”的第一步映射。

虽然这里只是改一个角色说明，但它已经让最小 hello agent 开始朝主案例方向靠拢。

### 可能出现的问题

- 修改了提示词但结果变化不明显：说明输入过短或模型输出较随机
- 运行失败：优先排查环境，不要误以为是 prompt 改坏了
- 输出还是像普通聊天机器人：说明你现在只是完成角色引导，还没有加入工具、知识检索与流程

## 常见问题与排错

### 问题 1：`dotnet` 命令可执行，但 sample 无法构建
- 现象：`dotnet restore` 或 `dotnet run` 时报 SDK/TargetFramework/包恢复相关错误
- 原因：本机 SDK 版本与 `dotnet/global.json` 期望不匹配，或 NuGet 恢复环境异常
- 排查方法：
  - 查看 `dotnet/global.json`
  - 执行 `dotnet --list-sdks`
  - 确认是否具备 .NET 10 SDK
- 修复方式：安装匹配 SDK，或至少确保能够按仓库要求 roll-forward 到兼容小版本

### 问题 2：提示 `AZURE_OPENAI_ENDPOINT` 未设置
- 现象：运行时直接抛 `InvalidOperationException`
- 原因：sample 在入口代码中直接读取环境变量
- 排查方法：
  - 检查当前 PowerShell 会话里的 `$env:AZURE_OPENAI_ENDPOINT`
  - 检查是否在正确终端窗口设置过变量
- 修复方式：重新设置环境变量后，在同一终端内运行 `dotnet run`

### 问题 3：认证失败或 401/403
- 现象：请求模型时报鉴权失败
- 原因：
  - 没有 `az login`
  - 当前 Azure 账号没有目标资源权限
  - 登录了错误租户/错误账号
- 排查方法：
  - 执行 `az account show`
  - 检查 sample README 中要求的角色权限
  - 核对目标 Azure OpenAI 资源与当前账号是否一致
- 修复方式：
  - 重新 `az login`
  - 切换订阅/租户
  - 补齐资源角色授权

### 问题 4：sample 很多，不知道先看哪个
- 现象：目录越看越乱，迟迟不敢开始
- 原因：没有把 sample 当成“分层学习路线”，而是当成平铺文件夹
- 排查方法：直接回到 `dotnet/samples/README.md`
- 修复方式：严格按 `01-get-started -> 02-agents -> 03-workflows -> 04-hosting -> 05-end-to-end` 递进

### 问题 5：误把 sample 代码当成框架源码
- 现象：看到 `Program.cs` 就以为这就是框架内部实现
- 原因：没有建立 `samples` 与 `src` 的边界意识
- 排查方法：看当前文件是否位于 `dotnet/src` 还是 `dotnet/samples`
- 修复方式：记住：sample 是用法，src 才是实现

## 企业级落地提醒

### 1. 稳定性
开发环境和生产环境不要混为一谈。

本章里你大概率会使用：

- `DefaultAzureCredential` 或 Azure CLI 登录
- 本地环境变量
- 本地 PowerShell 会话

这些非常适合学习和验证，但不适合直接照搬到生产环境。企业里至少要进一步区分：

- 本地开发认证方式
- CI 构建认证方式
- 生产服务认证方式

否则“本地能跑”不等于“服务可上线”。

### 2. 可维护性
目录结构认知本身就是可维护性的一部分。

如果团队成员都不清楚：

- 样例在什么地方
- 核心能力在哪个项目里
- 测试要看哪一层

后续扩展 Agent 系统时，阅读与协作成本会迅速上升。

建议团队尽早形成统一口径：

- 先用 `samples` 选能力入口
- 再到 `src` 找实现与扩展点
- 再到 `tests` 补边界条件

### 3. 安全与权限边界
本章最容易被忽略的不是代码安全，而是**环境权限安全**。

例如：

- 用了错误的 Azure 账号
- 把生产资源 endpoint 配到本地测试里
- 在共享机器上保留敏感环境变量

这些问题看起来只是“环境配置”，但本质上已经属于权限边界问题。

### 4. 成本控制
Agent 开发从第二章就已经开始涉及成本。

只要 sample 连接了真实模型，就会产生：

- token 成本
- 资源调用成本
- 调试重试成本

所以从一开始就要养成习惯：

- 先跑最小 sample
- 先做最小修改
- 不要一上来跑端到端大样例
