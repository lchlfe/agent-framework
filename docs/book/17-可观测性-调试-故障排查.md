## 先建立整体认知

可观测性不是“多打印几行日志”。对生产级 Agent 来说，最少要同时回答四类问题：

1. **发生了什么**：日志负责记录关键事件，例如 RunAsync invoked、completed、failed、canceled。
2. **请求经过了哪里**：Trace/Activity 负责把用户请求、Agent 调用、Tool 调用、HTTP 调用、Workflow 节点串成一条链路。
3. **系统是否变慢或变贵**：Metrics 负责统计交互次数、耗时、token 使用量、错误率等聚合指标。
4. **能否复现和验证**：tests 与 sample verifier 负责把“看起来能跑”变成可重复验证的规则，例如输出中不应包含 error messages 或 stack traces。

在 Agent 系统中，这一层处于运行时治理层。缺少它时，常见后果是：

- 用户只反馈“回答错了/卡住了”，开发者无法还原当时的消息、工具、模型参数与状态。
- 多 Agent 或 Workflow 只看到最终失败，不知道失败发生在 Planner、Specialist、Tool 还是外部 API。
- 生产环境不能开启 Trace 级原文日志，否则可能把用户输入、函数参数、工具结果中的敏感信息写入遥测系统。
- 长任务失败后虽然有状态持久化，但没有 trace id 或 session id，无法把持久化记录与运行日志对应起来。

## 核心概念拆解

### 1. `LoggingAgent` 是什么

`LoggingAgent` 位于 `dotnet/src/Microsoft.Agents.AI/LoggingAgent.cs`，它是一个 `DelegatingAIAgent`，通过包装内部 `AIAgent` ���记录 `RunAsync` 和 `RunStreamingAsync` 的调用事件。

它的边界很清晰：

- Debug 级别：记录 invoked、completed、canceled、failed 等生命周期事件。
- Error 级别：记录调用失败及异常。
- Trace 级别：记录 messages、options、metadata、response、streaming update 等可能包含敏感信息的数据。

源码注释明确提醒：当 `ILogger` 启用 `LogLevel.Trace` 时，会记录消息、选项和响应内容，这些内容可能包含敏感应用数据；Trace 默认关闭，不应在生产环境随意开启。

### 2. `OpenTelemetryAgent` 是什么

`OpenTelemetryAgent` 位于 `dotnet/src/Microsoft.Agents.AI/OpenTelemetryAgent.cs`。它同样是 `DelegatingAIAgent`，但目标不是写普通日志，而是实现 OpenTelemetry Generative AI 语义约定相关的 Activity/span。

它内部通过 `OpenTelemetryChatClient` 和一个 `ForwardingChatClient` 把 Agent 调用转换为 chat client 调用链，再在当前 Activity 上补充 Agent 特有标签，例如：

- `gen_ai.operation.name = invoke_agent`
- `gen_ai.provider.name`
- `gen_ai.agent.id`
- `gen_ai.agent.name`
- `gen_ai.agent.description`
- `gen_ai.request.model`
- `gen_ai.response.id`
- `gen_ai.usage.input_tokens`
- `gen_ai.usage.output_tokens`

它还提供 `EnableSensitiveData`。默认情况下遥测包含 token、模型、provider 等元数据，但不包含原始输入输出。若设置为 true，或通过环境变量 `OTEL_INSTRUMENTATION_GENAI_CAPTURE_MESSAGE_CONTENT=true`，则可能捕获原始消息、函数参数、函数结果等敏感内容。

### 3. Workflow OpenTelemetry 是什么

Workflow 的可观测性入口在 `dotnet/src/Microsoft.Agents.AI.Workflows/OpenTelemetryWorkflowBuilderExtensions.cs`，配置对象在 `dotnet/src/Microsoft.Agents.AI.Workflows/Observability/WorkflowTelemetryOptions.cs`。

`WorkflowTelemetryOptions` 支持：

- `EnableSensitiveData`
- `DisableWorkflowBuild`
- `DisableWorkflowRun`
- `DisableExecutorProcess`
- `DisableEdgeGroupProcess`
- `DisableMessageSend`

这说明 Workflow 级别的观测不是只有一个总 span，而是可以覆盖构建、运行、Executor 处理、边处理、消息发送等不同层级。`ObservabilityTests.cs` 中的测试说明，一个简单的 uppercase -> reverse workflow 预期会产生 workflow.build、workflow.session、workflow_invoke、edge_group.process、executor.process、message.send 等活动。

### 4. sample 验证与 tests 扮演什么角色

`dotnet/eng/verify-samples/AgentsSamples.cs` 中把 `AgentOpenTelemetry` 登记为 sample，但标注了 SkipReason：需要 Aspire Dashboard / Docker 来收集 OpenTelemetry。因此它不是普通 CI 中无条件运行的 sample。

相关单元测试包括：

- `LoggingAgentTests.cs`：验证 Debug/Trace/Error/canceled/failed 的日志行为。
- `OpenTelemetryAgentTests.cs`：验证 Activity 标签、token usage、sensitive data 开关、是否有 listener 时是否产生活动。
- `OpenTelemetryAgentBuilderExtensionsTests.cs` 与 `LoggingAgentBuilderExtensionsTests.cs`：验证 builder 扩展能正确包装 agent。
- `ObservabilityTests.cs`：验证 workflow 级 Activity 的名称、数量、事件、敏感数据开关与禁用选项。

这些测试告诉我们：本章描述的能力不是单纯理论，而是框架源码已有并被测试覆盖的能力。

## 结合源码理解执行过程

建议按“入口 -> 中间处理 -> 关键对象 -> 输出结果”的顺序理解一次调用。

### 1. 入口在哪里

在 sample `dotnet/samples/02-agents/AgentOpenTelemetry/Program.cs` 中，入口包括三层：

1. OpenTelemetry 基础设施：
   - `Sdk.CreateTracerProviderBuilder()`
   - `Sdk.CreateMeterProviderBuilder()`
   - `serviceCollection.AddLogging(...).AddOpenTelemetry(...)`
2. chat client：
   - `AzureOpenAIClient(...).GetChatClient(deploymentName).AsIChatClient().AsBuilder().UseFunctionInvocation().UseOpenTelemetry(...)`
3. agent：
   - `new ChatClientAgent(...).AsBuilder().UseOpenTelemetry(...).Build()`

这意味着 sample 同时观测了：自定义应用 span、HTTP 调用、chat client 层、agent 层、metrics 和 logs。

### 2. 调用是如何继续流转的

一次交互的大致链路是：

```text
Console.ReadLine
  -> appLogger.LogInformation("Processing user interaction")
  -> activitySource.StartActivity("Agent Interaction")
  -> agent.RunStreamingAsync(userInput, session)
  -> OpenTelemetryAgent.RunCoreStreamingAsync
  -> OpenTelemetryChatClient.GetStreamingResponseAsync
  -> ForwardingChatClient.GetStreamingResponseAsync
  -> inner ChatClientAgent / instrumentedChatClient
  -> Azure OpenAI HTTP 调用或 Function Invocation
  -> streaming update 写回控制台
  -> metrics 记录 success/error 与 response time
  -> activity.SetStatus 或 SetTag
  -> appLogger.LogInformation / LogError
```

如果发生异常，sample 在 catch 中会：

- 在控制台输出 `Error: {ex.Message}`。
- 记录 error 状态的 counter 和 histogram。
- 在当前 Activity 上设置 `response.success=false`、`error.message` 和 `ActivityStatusCode.Error`。
- 使用 `appLogger.LogError(ex, ...)` 写结构化错误日志。

### 3. 关键对象分别负责什么

- `ActivitySource`：创建 `Agent Session` 和 `Agent Interaction` 这样的自定义 span，便于把业务会话与底层 Agent span 串起来。
- `Meter`：创建 `agent_interactions_total` 与 `agent_response_time_seconds`，用于观察吞吐与耗时。
- `ILogger`：写结构化日志，并通过 scope 携带 `SessionId`、`AgentName`。
- `OpenTelemetryAgent`：把 Agent 调用包装成 `invoke_agent` 语义的 span。
- `OpenTelemetryChatClient`：采集 chat client 层生成式 AI 语义标签，`OpenTelemetryAgent` 复用其实现。
- `LoggingAgent`：更适合开发期或排错期快速观察调用生命周期。
- `WorkflowTelemetryOptions`：控制 workflow build/run/executor/message 等 Activity 是否生成以及是否记录敏感输入输出。

### 4. 扩展点在哪里

- Agent 日志包装：通过 `LoggingAgentBuilderExtensions` 或直接 `new LoggingAgent(innerAgent, logger)`。
- Agent OpenTelemetry 包装：通过 `OpenTelemetryAgentBuilderExtensions`，sample 中使用 `.AsBuilder().UseOpenTelemetry(...).Build()`。
- Chat client OpenTelemetry 包装：通过 `ChatClientBuilder.UseOpenTelemetry(...)`。
- Workflow OpenTelemetry 包装：通过 `WorkflowBuilder.WithOpenTelemetry(...)`，并传入 `configure: opts => ...`。
- 自定义 trace：应用代码中创建自己的 `ActivitySource`，并在用户请求、工单 id、session id、审批 id 等业务边界上打 tag。

## 关键代码或配置讲解

### 代码片段 1：sample 中配置 Traces

```csharp
var tracerProviderBuilder = Sdk.CreateTracerProviderBuilder()
    .SetResourceBuilder(ResourceBuilder.CreateDefault().AddService(ServiceName, serviceVersion: "1.0.0"))
    .AddSource(SourceName)
    .AddHttpClientInstrumentation()
    .AddOtlpExporter(options => options.Endpoint = new Uri(otlpEndpoint));
```

- 作用：告诉 OpenTelemetry 采集哪个 `ActivitySource`，并把 HTTP client 调用也纳入 trace。
- 为什么重要：Agent 很多失败来自模型服务、网络、外部 API，仅观察 Agent span 不够。
- 改动它会影响什么：如果忘记 `.AddSource(SourceName)`，自定义 `Agent Session`、`Agent Interaction` 可能不会进入 Dashboard；如果没有 exporter，数据不会送到 Aspire Dashboard。

### 代码片段 2：sample 中同时包装 chat client 和 agent

```csharp
using var instrumentedChatClient = new AzureOpenAIClient(new Uri(endpoint), new DefaultAzureCredential())
    .GetChatClient(deploymentName)
        .AsIChatClient()
        .AsBuilder()
        .UseFunctionInvocation()
        .UseOpenTelemetry(sourceName: SourceName, configure: (cfg) => cfg.EnableSensitiveData = true)
        .Build();

var agent = new ChatClientAgent(instrumentedChatClient,
    name: "OpenTelemetryDemoAgent",
    instructions: "You are a helpful assistant that provides concise and informative responses.",
    tools: [AIFunctionFactory.Create(GetWeatherAsync)])
    .AsBuilder()
    .UseOpenTelemetry(sourceName: SourceName, configure: (cfg) => cfg.EnableSensitiveData = true)
    .Build();
```

- 作用：chat client 层和 agent 层都进入 OpenTelemetry。
- 为什么重要：chat client 更靠近模型请求，agent 层更靠近业务抽象；两层一起看，才能知道“是 Agent 编排慢，还是模型调用慢”。
- 改动它会影响什么：只包 agent 可能缺少 chat client 细节；只包 chat client 可能看不到 `invoke_agent` 和 agent metadata。

### 代码片段 3：`LoggingAgent` 的敏感信息边界

```csharp
if (this._logger.IsEnabled(LogLevel.Trace))
{
    this.LogInvokedSensitive(nameof(RunAsync), this.AsJson(messages), this.AsJson(options), this.AsJson(this.GetService<AIAgentMetadata>()));
}
else
{
    this.LogInvoked(nameof(RunAsync));
}
```

- 作用：Debug 下只记录调用生命周期；Trace 下记录消息、选项和元数据。
- 为什么重要：排错时常想看完整上下文，但生产环境必须保护 PII、工单内容、客户数据和工具结果。
- 改动它会影响什么：如果把敏感数据放到 Debug 或 Information，会显著增加泄露风险与日志成本。

### 代码片段 4：Workflow 遥测开关

```csharp
public bool DisableWorkflowBuild { get; set; }
public bool DisableWorkflowRun { get; set; }
public bool DisableExecutorProcess { get; set; }
public bool DisableEdgeGroupProcess { get; set; }
public bool DisableMessageSend { get; set; }
```

- 作用：控制 workflow 不同层级的 Activity 是否生成。
- 为什么重要：复杂 workflow span 数量可能很多，需要按排错场景选择粒度。
- 改动它会影响什么：关闭 executor 或 message span 会减少噪音和成本，但也可能让节点级故障难以定位。

## 做一个最小改造实验

### 实验目标

在不改变业务逻辑的前提下，让 `AgentOpenTelemetry` 更适合排查“某个工单请求为什么失败或变慢”。

### 改造点

在 sample 的交互循环里加入一个业务相关 tag，例如：

```csharp
var ticketId = $"TICKET-{interactionCount:0000}";
activity?
    .SetTag("ticket.id", ticketId)
    .SetTag("agent.name", "OpenTelemetryDemoAgent")
    .SetTag("interaction.number", interactionCount);

using (appLogger.BeginScope(new Dictionary<string, object>
{
    ["SessionId"] = sessionId,
    ["AgentName"] = "OpenTelemetryDemoAgent",
    ["TicketId"] = ticketId
}))
{
    // 调用 agent.RunStreamingAsync(...)
}
```

注意：本章只是给读者的实验建议，当前写作任务不修改 sample 源码。

### 你应该观察什么

- Aspire Dashboard 中 trace 是否出现 `ticket.id`。
- Structured Logs 是否能通过 `TicketId` 或 `SessionId` 过滤。
- 出错时同一个 ticket id 是否能关联到 LogError、error span 和 metrics status=error。

### 可能出现的问题

- 如果 Dashboard 中看不到自定义 tag，先确认 `ActivitySource` 是否被 `.AddSource(SourceName)` 注册。
- 如果 Structured Logs 中看不到 scope 字段，检查 `options.IncludeScopes = true` 是否设置。
- 如果 trace 中没有 HTTP 调用，检查是否启用了 `.AddHttpClientInstrumentation()`。

## 常见问题与排错

### 问题 1：Aspire Dashboard 没有 trace

- 现象：Console 正常回答，但 Dashboard 的 Traces 页为空。
- 原因：Dashboard 未启动、OTLP endpoint 配错、端口冲突、未配置 exporter、未 AddSource，或没有 Activity listener。
- 排查方法：
  1. 确认 Docker 容器存在：`docker ps`。
  2. 确认 Dashboard 地址：`http://localhost:4318`。
  3. 确认应用环境变量：`OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4317`。
  4. 检查 sample 中 `.AddOtlpExporter(...)` 和 `.AddSource(SourceName)` 是否存在。
- 修复方式：重启 Aspire Dashboard，释放 4317/4318 端口，确保应用和 Dashboard 使用同一 endpoint。

### 问题 2：只看到应用日志，看不到 Agent 内部 span

- 现象：Structured Logs 有记录，但 Traces 中没有 `invoke_agent` 或 chat client span。
- 原因：只配置了 logging provider，没有对 chat client 或 agent 调用 `.UseOpenTelemetry(...)`。
- 排查方法：检查 `Program.cs` 是否同时包含 chat client builder 的 `.UseOpenTelemetry(...)` 和 agent builder 的 `.UseOpenTelemetry(...)`。
- 修复方式：对需要观测的层级显式添加 OpenTelemetry wrapper；如果只想看 agent 层，也至少确认 `OpenTelemetryAgentBuilderExtensions` 已生效。

### 问题 3：Debug 日志有 invoked/completed，但看不到消息内容

- 现象：日志只显示 `RunAsync invoked`、`RunAsync completed`。
- 原因：`LoggingAgent` 只有在 `LogLevel.Trace` 启用时才记录 messages、options、response。
- 排查方法：查看 `LoggingAgent.cs` 中 Debug 与 Trace 分支，确认当前 logger 最小级别。
- 修复方式：开发环境可临时启用 Trace；生产环境不要直接启用原文 Trace，应使用脱敏字段、采样和受控重放机制。

### 问题 4：OpenTelemetry 中没有原始输入输出

- 现象：span 有 token、model、provider，但没有 `gen_ai.input.messages` 或 `gen_ai.output.messages`。
- 原因：`EnableSensitiveData` 默认为 false，或环境变量未开启。
- 排查方法：参考 `OpenTelemetryAgentTests.cs`，当 `EnableSensitiveData=false` 时测试明确断言不包含输入输出消息。
- 修复方式：只在本地或受控排错环境设置 `cfg.EnableSensitiveData = true`；生产中需要评估 PII、访问控制、保留周期与审计。

### 问题 5：Workflow trace 太多或太少

- 现象：复杂 workflow 中 span 数量过多，或看不到 executor/message 级别信息。
- 原因：`WorkflowTelemetryOptions` 的禁用项配置过粗或过细。
- 排查方法：检查是否设置了 `DisableExecutorProcess`、`DisableEdgeGroupProcess`、`DisableMessageSend` 等选项。
- 修��方式：平时保留 workflow run 和关键 executor 级 span；深度排错时临时开启 message send 和 sensitive data；高流量生产环境结合采样与过滤。

### 问题 6：Azure OpenAI 认证失败

- 现象：sample 启动或调用时报 endpoint/credential/deployment 相关错误。
- 原因：`AZURE_OPENAI_ENDPOINT` 未设置、deployment 名称错误、`az login` 未登录或账号无权限。
- 排查方法：检查环境变量、执行 `az account show`、确认 Azure OpenAI deployment 名称。
- 修复方式：重新设置环境变量，执行 `az login`，或在生产中改用更明确的 Managed Identity 配置，避免 `DefaultAzureCredential` 在多种凭据之间探测导致延迟和误用。

## 本章在综合案例中的增量

主案例仍然是“企业内部知识与工单协同 Agent 系统”。在第 16 章之后，它已经具备服务化部署与长任务运行基础；本章为它新增“生产可观测与排错闭环”。

新增能力包括：

- 在用户入口生成或传入 `session.id`、`ticket.id`、`trace.id`。
- 在 Router/Planner/工单 Specialist/知识检索/外部工单工具调用之间传播 Activity 上下文。
- 对 Agent 层启用 `UseOpenTelemetry`，记录 `invoke_agent`、provider、model、token usage、response id。
- 对 Workflow 层启用 `WithOpenTelemetry`，观察工单流程的 build/run/executor/message 层级。
- 对日志启用结构化字段和 scope，确保一条失败工单能从日志跳到 trace，再跳到持久化状态或评测样本。
- 对敏感内容采用默认关闭策略：默认只记录元数据；只有在受控排错环境中短时间开启 `EnableSensitiveData` 或 Trace 级日志。

如果不加入这一章的能力，主案例上线后会卡在这些问题上：

- 工单处理失败只能看到最终错误，无法知道是知识检索无结果、Tool 参数错误、审批超时还是模型服务失败。
- 多轮会话回答不一致时，无法定位当时的 Prompt、上下文压缩、工具结果或状态恢复是否异常。
- 成本异常升高时，无法按 agent、workflow、ticket 类型统计 token 与耗时。
- 第 18 章要做评测回归时，缺少失败样本来源和 trace/log 证据链。

与仓库已有源码的关系：

- 仓库已有 sample 直接支持：`AgentOpenTelemetry` 已展示 Aspire Dashboard、OTLP、Application Insights 可选导出、agent/chat client OpenTelemetry、日志和 metrics。
- 可以基于已有 sample 做最小改造实现：增加 `ticket.id`、`conversation.id`、`approval.id` 等业务 tag 和 logging scope。
- 仓库未直接给出现成样例但可基于框架扩展：完整企业工单系统的跨服务 trace、RAG 检索指标、审批链路审计与成本 dashboard。

## 企业级落地提醒

- 稳定性：可观测性不能成为主链路风险点。Exporter 或 Dashboard 不可用时，业务调用应尽量降级而不是整体失败；高流量场景要配置采样。
- 可维护性：统一字段命名，例如 `session.id`、`ticket.id`、`agent.name`、`workflow.id`，不要每个模块各自发明字段。
- 权限边界：Trace/Log 系统通常被更多运维人员访问，不应默认写入用户原文、客户数据、工单附件、凭据或工具返回的敏感结果。
- 安全风险：`EnableSensitiveData=true`、`LogLevel.Trace`、`OTEL_INSTRUMENTATION_GENAI_CAPTURE_MESSAGE_CONTENT=true` 都应视为高风险开关，必须限制环境、时间、访问人群和保留周期。
- 成本控制：span、metrics、log 都会产生存储和查询成本；复杂 Workflow 的 message 级 span 在高并发下尤其要谨慎。
- 可测试性：将排错经验沉淀为测试。仓库中的 `LoggingAgentTests`、`OpenTelemetryAgentTests`、`ObservabilityTests` 提供了很好的范式：不要只靠人工看 Dashboard。
- 持久化与恢复：长任务系统要把 trace id/session id/run id 写入状态或业务记录，失败恢复时才能串回原始调用链。

## 本章小结

- 本章最重要的结论 1：Agent 可观测性至少由日志、Trace、Metrics、测试验证共同构成，单靠 Console.WriteLine 不能支撑生产排错。
- 本章最重要的结论 2：仓库已经提供 `LoggingAgent`、`OpenTelemetryAgent`、Workflow OpenTelemetry 与 `AgentOpenTelemetry` sample，应该优先基于这些真实入口学习和扩展。
- 本章最重要的结论 3：敏感数据默认不应进入日志和遥测；只有在受控排错环境中才考虑临时开启 Trace 或 `EnableSensitiveData`。

下一章将讨论评测体系与质量保障。本章的 trace、日志、失败样本和运行指标，会成为第 18 章做正确率、稳定性、回归质量和业务效果评估的重要输入。
