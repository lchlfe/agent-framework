# Microsoft Agent Framework for .NET 学习手册

> 面向希望从入门逐步成长为资深 Agent 系统开发者的 .NET 开发者。
>
> 目标：从“看不懂仓库、跑不通 sample”，到“能独立开发、设计、部署和演进大型应用级 Agent 系统”。

## 使用方式

- 先读本文件，建立全局地图
- 再按阶段推进：入门 → 中级 → 高级 → 资深
- 每章都尽量配合 `dotnet/samples` 实操
- 每完成一个阶段，最好做一个阶段性项目
- 后续可以继续在每章下补充示例、笔记、源码索引

---

## 成长路径

### 入门
目标：会跑、会改、会解释最小 Agent 的基本工作方式。

### 中级
目标：能独立做出一个可用的 Agent 应用原型，理解状态、编排和多 Agent 协作。

### 高级
目标：能设计复杂业务场景下的 Agent 系统，处理知识、工具集成和人工介入。

### 资深
目标：能把 Agent 系统做成可上线、可观测、可评测、可治理、可持续演进的生产级平台。

## 章节编写规范

为了支持多个对话并行撰写不同章节，同时尽量避免遗漏内容，后续每一章都应遵循同一套写作规范。

### 1. 编写目标

每一章都必须同时满足以下目标：

- 让读者知道“这一章为什么学”
- 让读者知道“这一章对应仓库里的哪些真实内容”
- 让读者能亲手运行、修改、验证至少一个相关 sample 或代码片段
- 让读者知道“这一章的能力在大型 Agent 应用里处于什么位置”
- 让读者知道“这一章有哪些常见遗漏点、风险点和边界条件”

### 2. 章节必须包含的固定结构

除非有明确特殊原因，每一章都应至少包含以下部分，并尽量保持顺序一致：

1. **本章目标**
   - 说明本章要解决什么问题
   - 说明学完后具体能做到什么

2. **你会先获得什么认知**
   - 用简短语言说明本章在整体学习地图中的位置
   - 说明它和前后章节的关系

3. **源码与目录定位**
   - 明确指出本章对应的仓库目录、项目、类、接口或 sample
   - 如果一个主题横跨多个目录，要明确主次关系

4. **核心概念拆解**
   - 不只给名词，要解释职责、边界、关系、执行流程
   - 尽量用“是什么、为什么、怎么协作、什么时候会出问题”的方式展开

5. **实际运行或验证步骤**
   - 给出可执行的 sample、命令、配置步骤
   - 说明运行前提、预期现象、成功标志

6. **结合源码解释执行过程**
   - 说明一次调用是如何流转的
   - 标出关键入口、关键对象、关键扩展点

7. **最小改造实验**
   - 至少给出一个“小改一下就能观察变化”的实验任务
   - 例如：改 Prompt、加 Tool、改状态存储、插入日志、调整工作流节点

8. **常见问题与排错**
   - 说明本章最常见的报错、误解、遗漏点
   - 说明如何定位问题，不只给结论

9. **企业级落地提醒**
   - 说明本章内容在真实企业项目中的注意事项
   - 至少覆盖稳定性、可维护性、安全性或成本中的一个方面

10. **本章小结**
    - 总结本章结论
    - 明确下一章为什么要学

### 3. 每章必须覆盖“仓库已有内容”

每一章都不应只写概念，还必须主动覆盖仓库里已经存在的内容。至少要做到：

- 明确本章相关的 sample 在哪里
- 明确本章相关的项目或源码目录在哪里
- 明确本章相关的关键类型、接口、扩展点或配置点在哪里
- 明确哪些内容是仓库已支持的，哪些只是理论延伸
- 如果仓库中存在多个实现路径，要指出推荐先看哪一个

如果当前仓库里没有直接对应的完整示例，也要明确写出：

- 仓库中最接近的参考入口是什么
- 这一章有哪部分需要基于现有源码做推导理解
- 哪些能力是框架思路具备、但仓库示例未完整展开的

### 4. 每章必须回答的检查问题

为减少遗漏，每一章在完成前都至少自查以下问题：

- 这一章是否明确说明了“要解决的问题”
- 这一章是否明确指出了“仓库中的对应源码位置”
- 这一章是否至少引用了一个 sample、项目、类或接口
- 这一章是否解释了核心执行链路，而不只是概念定义
- 这一章是否给了读者可操作的实验或验证方式
- 这一章是否说明了常见误区、失败点或边界条件
- 这一章是否说明了企业级落地时需要额外关注什么
- 这一章是否避免了脱离源码的空泛总结

如果以上任一问题回答是否定的，该章节就不算完成。

### 5. 多对话并行撰写时的统一要求

因为这 20 个章节可能由多个对话并行完成，所以每一章都必须遵守以下统一要求：

- 术语保持一致：同一个概念不要在不同章节里反复换名字
- 结构保持一致：尽量使用相同的小节模板
- 深度保持一致：不能有的章节只写概念，有的章节只贴代码
- 口径保持一致：明确区分“框架已提供”“sample 已展示”“作者补充建议”
- 引用保持一致：涉及源码时尽量写清楚目录、项目名、类型名
- 边界保持一致：不要把下一章应重点展开的内容在本章写得过深，避免章节互相侵占

### 6. 写作边界与取舍原则

每一章都应遵守以下边界原则：

- 不要只讲理论，必须回到仓库源码和 sample
- 不要只贴代码，必须解释为什么这样设计
- 不要把一章写成整个框架总览，重点解决本章核心问题
- 不要为了完整而堆砌无关背景，优先保证可执行、可验证、可定位
- 不要假设读者已经知道隐藏前提，关键依赖条件必须写明

### 7. 推荐输出物

每一章完成后，最好至少留下以下产物中的若干项：

- 一段本章核心理解
- 一次 sample 运行记录
- 一张调用链或结构图
- 一个最小改造实验
- 一个问题清单
- 一个架构决策记录

### 8. 完成判定标准

只有同时满足以下条件，该章节才算真正完成：

- 已覆盖章节主题对应的仓库已有内容
- 已给出至少一个可执行或可验证的实践步骤
- 已解释关键概念、关键流程和关键源码入口
- 已补充常见问题、遗漏点或企业级注意事项
- 已形成可复用的章节结构，方便和其他章节保持一致

## 综合案例设计规范

为了避免 20 个章节分别成文却彼此割裂，整套手册应围绕一个统一的企业级 Agent 应用主线案例逐步展开。后续每一章除了讲清本章知识点外，还应明确说明“这一章在综合案例里新增了什么能力”。

### 1. 为什么必须有综合案例

如果没有综合案例，章节写作很容易出现以下问题：

- 每章各讲各的，读者知道概念，但不知道如何拼成系统
- 单章示例能跑，但无法理解能力如何逐步演进为完整应用
- 多个对话并行写作时，容易出现案例背景不一致、命名不一致、边界不一致
- 最终读者学完 20 章后，仍然缺少一个“从最小原型到企业级系统”的完整认知闭环

因此，这套手册应至少有一个贯穿全书的综合案例，并满足：

- 能覆盖单 Agent、Tool、Workflow、多 Agent、Memory、RAG、审批、部署、评测、安全等核心能力
- 能从简单版本逐步演进，不需要一开始就把所有能力堆满
- 能映射到仓库已有 sample、项目结构和扩展点
- 能代表一个真实企业应用，而不只是教学性质的玩具案例

### 2. 综合案例的推荐形态

综合案例应优先选择“企业内部可落地、边界清晰、能力覆盖全面”的场景。建议优先采用以下类型之一：

- 企业知识助理 + 工单处理 Agent
- 企业内部运维/支持协作 Agent
- 面向业务流程的任务分发与审批 Agent
- 结合知识检索、工具调用和人工审批的综合业务助手

如果后续没有特殊业务约束，建议整套书默认采用如下主线：

- **主案例：企业内部知识与工单协同 Agent 系统**

这个主案例适合覆盖：

- 第 3-5 章：最小 Agent、Prompt、Tool 调用
- 第 7-10 章：会话、上下文、工作流、状态持久化
- 第 11-15 章：系统分层、RAG、外部系统集成、审批、任务拆解
- 第 16-19 章：服务化部署、观测、评测、安全与治理
- 第 20 章：从 sample 到大型应用的演进路线

### 3. 综合案例必须具备的组成部分

无论采用什么业务主题，综合案例都应尽量具备以下组成部分：

1. **用户入口**
   - 用户如何发起请求
   - 请求类型有哪些

2. **Agent 决策层**
   - 如何判断是直接回答、调用工具、检索知识还是进入工作流

3. **知识层**
   - 至少包含知识检索或文档查询能力

4. **工具层**
   - 至少包含 2-3 类不同副作用级别的工具
   - 例如只读查询工具、写操作工具、外部系统集成工具

5. **流程层**
   - 至少包含一个多步骤任务流程
   - 例如问题分类 → 检索 → 生成建议 → 调用系统 → 人工确认 → 返回结果

6. **状态层**
   - 能说明会话状态、任务状态、持久化和恢复策略

7. **人工介入层**
   - 至少有一个高风险动作需要人工确认或审批

8. **治理层**
   - 至少体现日志、评测、安全或成本治理中的若干项

### 4. 综合案例的章节演进要求

综合案例不应一次性写完，而应随着章节推进逐步增强。建议采用“阶段递进式”设计：

#### 第一阶段：最小可运行原型
对应章节：第 3-5 章

目标：先让主案例具备最小交互闭环。

至少包括：

- 一个最小 Agent
- 一段明确的系统 Prompt
- 至少一个简单 Tool
- 一次最小请求到响应的执行链路

#### 第二阶段：升级为可用应用原型
对应章节：第 7-10 章

目标：让主案例从单轮交互升级为有状态、有流程的应用。

至少包括：

- 会话与上下文管理
- 至少一个多步骤 Workflow
- 至少一个持久化状态点
- 失败后恢复或续跑的设计说明

#### 第三阶段：升级为复杂业务系统
对应章节：第 11-15 章

目标：让主案例从原型升级为复杂业务架构。

至少包括：

- 明确的系统分层
- 知识库或 RAG 接入
- 外部系统或 MCP 集成
- 人工审批/人工确认节点
- 任务拆解和职责划分机制

#### 第四阶段：升级为生产级服务
对应章节：第 16-19 章

目标：让主案例具备上线运行与持续治理能力。

至少包括：

- Hosting 与服务化部署形态
- 日志、链路追踪、调试方案
- 评测和回归机制
- 安全、权限、合规、成本控制策略

### 5. 每一章都必须说明“本章对综合案例新增了什么”

为避免章节孤立，每一章都应增加一个固定小节，例如：

- **本章在综合案例中的增量**

这个小节至少回答：

- 本章之前，综合案例处于什么状态
- 本章加入了哪一个新能力
- 这个能力修改了哪些模块、流程或边界
- 加入后，系统能力发生了什么变化
- 如果不加入这一章的能力，综合案例会卡在哪里

### 6. 综合案例必须保持命名与边界一致

多个对话并行写作时，综合案例必须统一以下内容：

- 统一案例名称
- 统一业务背景
- 统一角色名称
- 统一模块命名
- 统一工具名称
- 统一外部系统名称
- 统一状态对象命名

不要在不同章节中把同一个案例写成多个不同系统。例如：

- 前面写“知识助理”
- 后面写“审批机器人”
- 再后面又变成“工单系统 Copilot”

可以演进，但必须写清楚它们是同一主案例在不同阶段的扩展，而不是换了一个案例。

### 7. 综合案例与仓库已有源码的关系要求

综合案例不能完全脱离仓库现实，必须明确区分三类内容：

1. **仓库已有 sample 直接支持的部分**
2. **可以基于已有 sample 做最小改造实现的部分**
3. **仓库未直接给出现成样例、但可基于框架能力扩展出来的部分**

每一章写综合案例时，都应明确说明当前新增能力属于哪一类。

### 8. 综合案例的最终交付物要求

当 20 章全部完成后，整套书应至少能沉淀出以下综合成果：

- 一份完整的主案例架构说明
- 一份从最小原型到生产级系统的演进路线图
- 一份核心模块关系图或调用链图
- 一份关键状态与数据流说明
- 一份主要风险点与治理策略清单
- 一份可继续扩展为真实项目的实现蓝图

### 9. 写作时的最低约束

如果某一章没有显式说明它与综合案例的关系，该章节视为未完全完成。

如果某一章引入了新的案例设定，却没有说明与主案例的关系，也视为未完全完成。

- 能部署生产级 Agent 服务
