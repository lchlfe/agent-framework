# 第 19 章：安全、权限、合规与成本治理

> 本章定位：把前面已经能运行、能部署、能评测的 Agent 系统，补上生产环境必须具备的风险边界、访问控制、合规拦截、审计与成本约束。

- 理解 Purview 集成在 Prompt/Response 前后做合规处理的调用链。
- 使用审批、函数调用中间件、上下文压缩和限额策略，为综合案例补齐生产级治理层。

## 先建立整体认知

安全、权限、合规与成本治理不是一个单独的类，也不是只靠一句 Prompt 就能完成的能力。它们应该分布在 Agent 系统的多个层次：

- **入口层**：验证用户身份、租户、scope、角色、来源和请求大小。
- **Agent 层**：限制系统指令、隔离上下文、处理 Prompt Injection、记录会话与操作意图。
- **Tool 层**：按工具进行最小权限校验，区分只读工具、写操作工具、高风险工具。
- **审批层**：对支付、删除、审批、发邮件、改配置等副作用操作引入人工确认。
- **合规层**：在输入和输出两侧做内容处理、敏感信息识别、策略阻断与审计。
- **成本层**：控制 token、轮次、工具调用次数、模型选择、摘要/压缩策略和任务超时。

如果缺少这一层，企业内部知识与工单协同 Agent 系统会出现几个典型问题：普通员工可以让 Agent 执行经理权限的审批；恶意用户通过 Prompt Injection 要求模型忽略系统规则；检索到的内部文档被原样输出给无权限用户；长对话持续膨胀导致模型费用和延迟失控；生产事故发生后无法回答“谁在什么时间让 Agent 调用了哪个工具”。

## 先跑通或先验证一个最小示例

### 示例目标

验证同一个 Agent 服务中：

- `/chat` 端点需要 `agent.chat` scope。
- `ListPendingExpenses` 工具需要 `expenses.view` scope。
- `ApproveExpense` 工具需要 `expenses.approve` scope。
- 即使用户有审批 scope，也不能超过业务金额上限 €1,000。

### 运行前提

- 已安装 Docker 和 Docker Compose。
- 已有 OpenAI 兼容模型访问凭据。
- 在仓库根目录 `D:\code\agent-framework` 下执行命令。

### 操作步骤

1. 设置环境变量：

   ```bash
   export OPENAI_API_KEY="<your-openai-api-key>"
   export OPENAI_MODEL="gpt-5.4-mini"
   ```

   Windows PowerShell 可使用：

   ```powershell
   $env:OPENAI_API_KEY="<your-openai-api-key>"
   $env:OPENAI_MODEL="gpt-5.4-mini"
   ```

2. 启动授权 sample：

   ```bash
   cd dotnet/samples/05-end-to-end/AspNetAgentAuthorization
   docker compose up
   ```

3. 打开 `http://localhost:8080`，分别使用两个内置用户登录：

   - `testuser` / `password`：拥有 `agent.chat`、`expenses.view`、`expenses.approve`。
   - `viewer` / `password`：拥有 `agent.chat`、`expenses.view`，没有 `expenses.approve`。

4. 分别输入：

   - `Show me the pending expenses`
   - `Approve expense #1`
   - `Approve expense #3`

### 预期现象

- 两个用户都能查看 pending expenses。
- `viewer` 审批时会被拒绝，因为没有 `expenses.approve`。
- `testuser` 可以审批 #1，因为金额 €850 未超过上限。
- `testuser` 审批 #3 仍然会被拒绝，因为金额 €4,500 超过 `ApprovalLimit = 1000m`。

### 成功标志

成功标志不是“模型回答看起来合理”，而是权限边界被服务端代码强制执行：

- `/chat` 没有 token 或没有 `agent.chat` 时不能访问。
- 工具内部根据 `IUserContext.Scopes` 拒绝无权限操作。
- 业务限额由 `ExpenseService.ApproveExpense` 判断，而不是只写在 Prompt 里。

## 核心概念拆解

### 1. Prompt Injection 是什么

Prompt Injection 是指用户、检索文档、网页、工单描述或工具返回结果中包含“忽略之前规则”“泄露系统提示词”“调用某个高风险工具”等恶意指令，诱导模型绕过原本的系统约束。

在 Agent 系统里，Prompt Injection 的危险点不只是“回答错”，而是模型可能进一步调用工具执行副作用动作。因此治理原则是：

- Prompt 可以提醒模型，但不能作为唯一安全边界。
- 高风险工具必须在服务端二次校验权限、参数、业务限额和审批状态。
- RAG 文档、网页内容、用户输入要被视为不可信数据，而不是高优先级指令。
- 工具返回的内容如果要继续进入上下文，也要考虑脱敏、摘要和隔离。

仓库没有提供一个名为 `PromptInjectionDetector` 的完整组件，但可以利用工具级授权、审批、Purview、函数中间件和评测用例共同降低风险。

### 2. 权限边界是什么

权限边界至少分三层：

- **端点级权限**：用户能不能访问 Agent 服务。`AspNetAgentAuthorization/Service/Program.cs` 中 `/chat` 使用 `[Authorize(Policy = "AgentChat")]`，策略要求 `agent.chat` scope。
- **工具级权限**：用户能不能执行某个能力。`ExpenseService.ListPendingExpenses` 检查 `expenses.view`，`ApproveExpense` 检查 `expenses.approve`。
- **业务级权限**：即使有工具权限，是否满足业务限制。`ApproveExpense` 中 `ApprovalLimit = 1000m`，超过金额直接拒绝。

这三个边界不能互相替代。只做端点权限，登录用户仍可能让 Agent 调用超出其岗位范围的工具；只做 Prompt 约束，恶意输入可以诱导模型忽略规则；只做前端隐藏按钮，直接请求 API 仍可能绕过。

### 3. 合规治理是什么

合规治理关注输入和输出内容是否符合组织策略，例如敏感数据、受保护文档、DLP、审计活动和策略动作。

`AgentWithPurview/Program.cs` 展示了最小接入方式：

```csharp
using IChatClient client = new AzureOpenAIClient(
    new Uri(endpoint),
    new DefaultAzureCredential())
    .GetResponsesClient()
    .AsIChatClient(deploymentName)
    .AsBuilder()
    .WithPurview(browserCredential, new PurviewSettings("Agent Framework Test App"))
    .Build();
```

在源码中，`PurviewAgent.RunCoreAsync` 会调用 `PurviewWrapper.ProcessAgentContentAsync(...)`。`PurviewWrapper` 对 prompt 侧调用 `ProcessMessagesAsync(..., Activity.UploadText, ...)`，如果策略阻断则返回 `BlockedPromptMessage`；模型响应生成后，再对 response 侧调用 `ProcessMessagesAsync(..., Activity.DownloadText, ...)`，如果阻断则返回 `BlockedResponseMessage`。

这说明合规治理不是只检查输入，也要检查输出。

### 4. 成本治理是什么

成本治理包括 token、模型调用次数、工具调用次数、长任务时长、检索范围和上下文历史长度。仓库中最直接的成本控制样例是 `Agent_Step18_CompactionPipeline`：

- `ToolResultCompactionStrategy`：压缩旧工具调用结果。
- `SummarizationCompactionStrategy`：用模型摘要较早的对话片段。
- `SlidingWindowCompactionStrategy`：只保留最近 N 个用户轮次。
- `TruncationCompactionStrategy`：作为紧急 token budget 后备策略。
- `CompactionTriggers.TokensExceed(...)`、`MessagesExceed(...)`、`TurnsExceed(...)`：定义何时触发压缩。

成本治理的原则是：在服务端设置硬边界，在 Agent 层做上下文策略，在观测层记录每次调用的用量。

### 5. 常见误解是什么

- 误解一：系统 Prompt 写了“不要越权”就安全。实际工具仍必须服务端校验。
- 误解二：用户登录了就能共享同一个 Agent 单例的所有状态。实际要谨慎处理 session、工具依赖生命周期和多用户隔离。
- 误解三：合规只要检查用户输入。实际模型输出也可能泄露敏感信息。
- 误解四：成本控制等于换便宜模型。实际还需要限制上下文、压缩历史、减少不必要工具调用和设置任务超时。
- 误解五：审批就是让模型问一句“是否确认”。真正的审批结果必须作为结构化输入返回，并由工具调用链验证。

## 结合源码理解执行过程

### 1. 入口在哪里

权限主线入口在 `dotnet/samples/05-end-to-end/AspNetAgentAuthorization/Service/Program.cs`：

```csharp
builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options => { ... });

builder.Services.AddAuthorizationBuilder()
    .AddPolicy("AgentChat", policy =>
        policy.RequireAuthenticatedUser()
              .RequireAssertion(context => { ... }));

app.MapPost("/chat", [Authorize(Policy = "AgentChat")] async (..., AIAgent agent) =>
{
    var response = await agent.RunAsync(request.Message);
    return Results.Ok(new ChatResponse(response.Text, userContext.DisplayName));
});
```

这里先通过 JWT Bearer 验证身份，再用 `AgentChat` policy 检查 `scope` claim 中是否包含 `agent.chat`，最后才调用 `agent.RunAsync(...)`。

### 2. 调用是如何继续流转的

`AIAgent` 被注册为 DI 单例，并通过 `OpenAIClient(apiKey).GetChatClient(model).AsAIAgent(...)` 创建。它的工具来自：

```csharp
AIFunctionFactory.Create(expenseService.ListPendingExpenses),
AIFunctionFactory.Create(expenseService.ApproveExpense)
```

当模型决定调用工具时，真正执行的是 `ExpenseService` 方法。工具方法内部再读取 `IUserContext.Scopes`：

```csharp
if (!this._userContext.Scopes.Contains("expenses.approve"))
{
    return "Access denied. You do not have the expenses.approve scope.";
}
```

这就是“模型建议调用工具”和“服务端允许执行工具”的分离。

### 3. 关键对象分别负责什么

- `KeycloakUserContext`：从当前 `HttpContext.User` 解析 `sub`、`preferred_username`、`scope` 等 claim，并缓存到 `HttpContext.Items`。
- `ExpenseService`：承载业务数据、工具方法、scope 校验和金额上限。
- `AIAgent`：根据用户消息和工具定义生成响应或调用工具。
- `ApprovalRequiredAIFunction`：把普通工具包装为需要人类审批的工具，返回 `ToolApprovalRequestContent`。
- `PurviewWrapper`：在 prompt 和 response 两侧调用 Purview 内容处理逻辑。
- `CompactionProvider`：作为 `AIContextProvider` 管理上下文压缩，降低 token 膨胀。

### 4. 扩展点在哪里

- 如果要增加权限维度，可以扩展 `IUserContext`，加入 `TenantId`、`Department`、`Roles`、`ApprovalLimit`。
- 如果要统一工具授权，可以通过函数调用中间件扩展。`FunctionInvocationDelegatingAgent` 会把 `ChatOptions.Tools` 中的 `AIFunction` 包装成 `MiddlewareEnabledFunction`，在 `InvokeCoreAsync` 前后执行自定义逻辑。
- 如果要加强合规，可以在 `.WithPurview(...)` 外再叠加日志、脱敏、拒答模板或事件审计。
- 如果要控制成本，可以调整 `CompactionTriggers` 阈值、摘要模型、滑动窗口大小，并在服务层增加每用户/每租户调用预算。

## 关键代码或配置讲解

### 代码片段 1：端点级授权

位置：`dotnet/samples/05-end-to-end/AspNetAgentAuthorization/Service/Program.cs`

- 作用：通过 ASP.NET Core Authentication/Authorization 确保只有带 `agent.chat` scope 的用户能调用 `/chat`。
- 为什么重要：它阻止未授权用户访问 Agent 入口，是最外层边界。
- 改动它会影响什么：如果去掉 `[Authorize(Policy = "AgentChat")]`，任何能访问服务的人都可能触发 Agent 运行和后续工具调用。

### 代码片段 2：工具级授权与业务限额

位置：`dotnet/samples/05-end-to-end/AspNetAgentAuthorization/Service/ExpenseService.cs`

- 作用：在 `ListPendingExpenses` 和 `ApproveExpense` 内部检查 scope；审批时还检查 `ApprovalLimit`。
- 为什么重要：这是防止 Prompt Injection 和模型误调用的关键边界，因为权限由服务端代码强制执行。
- 改动它会影响什么：如果把 scope 检查只写到 Prompt 中，模型一旦被诱导或误判，工具就可能执行越权操作。

### 代码片段 3：Purview 输入/输出处理

位置：`dotnet/src/Microsoft.Agents.AI.Purview/PurviewWrapper.cs`

- 作用：在调用内部 Agent 或 ChatClient 前处理 prompt；在得到响应后处理 response。
- 为什么重要：合规阻断需要覆盖双向数据流。输入可能包含不合规内容，输出也可能泄露受保护信息。
- 改动它会影响什么：如果只检查输入，不检查输出，就无法阻止模型把敏感上下文、检索内容或工具结果返回给用户。

### 代码片段 4：Compaction 成本控制

位置：`dotnet/samples/02-agents/Agents/Agent_Step18_CompactionPipeline/Program.cs`

- 作用：使用 `PipelineCompactionStrategy` 组合工具结果压缩、摘要、滑动窗口和截断策略。
- 为什么重要：长会话和工具调用结果会快速放大 token，导致成本和延迟上升。
- 改动它会影响什么：阈值过低可能丢失上下文；阈值过高可能成本失控；摘要模型过贵也可能抵消压缩收益。

## 做一个最小改造实验

### 实验目标

在 `AspNetAgentAuthorization` sample 中增加一个更严格的业务规则：只有提交人不是当前用户时才允许审批，避免“自批”。

### 改造点

只作为本章练习建议，不要求提交到仓库。修改 `dotnet/samples/05-end-to-end/AspNetAgentAuthorization/Service/ExpenseService.cs` 的 `ApproveExpense`：

```csharp
if (string.Equals(expense.Submitter, this._userContext.DisplayName, StringComparison.OrdinalIgnoreCase) ||
    string.Equals(expense.Submitter, this._userContext.UserName, StringComparison.OrdinalIgnoreCase))
{
    return $"Cannot approve expense #{expenseId}. Users cannot approve their own expenses.";
}
```

为了更容易观察，你可以把某条 demo expense 的 `Submitter` 改成 `testuser` 或 `Test User`。

### 你应该观察什么

- 模型仍可能尝试调用 `ApproveExpense`。
- 工具代码会在服务端拒绝操作。
- Agent 最终应把工具返回的拒绝原因反馈给用户。

### 可能出现的问题

- 如果显示名和提交人字段格式不一致，规则可能无法命中。企业项目中应使用稳定的用户 ID，而不是显示名。
- 如果 `ExpenseService` 是单例但依赖了 per-request 状态，需要确认 `IUserContext` 的实现是否安全。sample 中 `KeycloakUserContext` 通过 `IHttpContextAccessor` 和 `HttpContext.Items` 按请求解析 claim，但真实项目中要评估服务生命周期。

## 本章在综合案例中的增量

主案例仍然是“企业内部知识与工单协同 Agent 系统”。在第 16-18 章之后，它已经具备服务化部署、可观测性和评测回归能力。本章新增的是治理层：

- **安全增量**：把用户输入、RAG 文档和工具返回都视为不可信内容；对高风险工具增加服务端校验与审批。
- **权限增量**：为用户入口增加认证；为工具增加 scope/role/tenant 检查；对写操作工具增加业务限额。
- **合规增量**：在用户请求进入 Agent 前、Agent 响应返回用户前接入合规处理，必要时阻断或替换响应。
- **成本增量**：对长会话启用压缩策略；设置每轮最大上下文、最大工具调用次数、预算和超时。
- **审计增量**：记录用户、会话、工具名、参数摘要、审批结果、策略阻断结果和 token 用量。

对应到模块变化：

```text
用户入口
  -> AuthN/AuthZ 中间件
  -> Agent Router
  -> Prompt/RAG 安全过滤
  -> Tool 执行前权限校验
  -> 高风险工具审批
  -> Tool 执行
  -> Purview/合规输出检查
  -> 成本与审计记录
  -> 返回用户
```

这些能力中，`AspNetAgentAuthorization`、`AgentWithPurview`、`ApprovalRequiredAIFunction`、`CompactionPipeline` 都有仓库 sample 可参考；跨租户权限模型、集中式预算看板、完整 Prompt Injection 分类器属于基于现有框架的企业扩展。

如果不加入本章能力，综合案例可以“能用”，但不能安全上线：工单工具可能越权，内部知识可能泄露，长会话成本不可控，审计无法满足企业要求。

## 常见问题与排错

### 问题 1：`viewer` 也能进入聊天界面，但审批失败

- 现象：`viewer` 登录后可以问 “Show me the pending expenses”，但 “Approve expense #1” 返回拒绝。
- 原因：`viewer` 有 `agent.chat` 和 `expenses.view`，没有 `expenses.approve`。
- 排查方法：查看 `keycloak/dev-realm.json` 中用户 scope，或在 `KeycloakUserContext.Scopes` 处打日志。
- 修复方式：如果这是预期权限，不需要修复；如果业务要求 viewer 也能审批，就需要给对应用户授予 scope，但不应绕过 `ExpenseService` 的校验。

### 问题 2：本地开发时 issuer 校验被关闭，是否安全

- 现象：`Program.cs` 中 `ValidateIssuer = !builder.Environment.IsDevelopment()`。
- 原因：Codespaces 或本地 Docker 中 Keycloak 的外部 URL 和容器内 URL 可能不一致，开发环境为了便于运行关闭 issuer 校验。
- 排查方法：检查 `ASPNETCORE_ENVIRONMENT`、`Auth:Authority`、token issuer。
- 修复方式：生产环境必须开启 issuer 校验、HTTPS metadata、audience 和 lifetime 校验，并使用正式 OIDC/Entra ID 配置。

### 问题 3：Purview sample 启动时报 `PURVIEW_CLIENT_APP_ID is not set`

- 现象：运行 `AgentWithPurview` 时报环境变量缺失。
- 原因：sample 需要 `AZURE_OPENAI_ENDPOINT`、`AZURE_OPENAI_DEPLOYMENT_NAME` 和 `PURVIEW_CLIENT_APP_ID`。
- 排查方法：检查环境变量和应用注册权限。
- 修复方式：设置对应变量，并确认 `InteractiveBrowserCredential` 使用的 Entra app 有调用 Purview API 的权限。生产中优先评估 `ManagedIdentityCredential` 等明确凭据。

### 问题 4：DefaultAzureCredential 在生产环境出现延迟或误用凭据

- 现象：启动或首次请求慢，或者使用了非预期身份访问 Azure OpenAI。
- 原因：多个 sample 都提示 `DefaultAzureCredential` 适合开发便利，但生产中可能有回退探测、延迟和潜在安全风险。
- 排查方法：打开 Azure.Identity 日志，确认实际命中的 credential 类型。
- 修复方式：生产中使用明确的凭据，例如 `ManagedIdentityCredential`，并限制托管身份权限。

### 问题 5：Compaction 后 Agent 忘记早期信息

- 现象：启用 `SlidingWindowCompactionStrategy` 或 `TruncationCompactionStrategy` 后，Agent 回答不了早期上下文。
- 原因：压缩或截断阈值过激，早期信息被摘要或删除。
- 排查方法：像 sample 中 `PrintChatHistory()` 一样观察消息数量，并记录 compaction 前后的关键内容。
- 修复方式：调高 `TurnsExceed` / `TokensExceed` 阈值；对关键业务状态使用结构化状态存储，不只依赖聊天历史。

### 问题 6：审批请求没有继续执行工具

- 现象：使用 `ApprovalRequiredAIFunction` 后，Agent 返回审批请求，但用户同意后没有继续执行。
- 原因：没有把 `ToolApprovalRequestContent.CreateResponse(true)` 作为用户消息传回同一个 `AgentSession`。
- 排查方法：对照 `Agent_Step01_UsingFunctionToolsWithApprovals/Program.cs` 中 while 循环处理 approvalRequests 的方式。
- 修复方式：保持会话，收集所有 `ToolApprovalRequestContent`，构造 `ChatMessage(ChatRole.User, [...])` 再调用 `agent.RunAsync(userInputResponses, session)`。

## 企业级落地提醒

- **稳定性**：安全和合规服务不可用时要明确 fail-open 还是 fail-closed。`PurviewSettings.IgnoreExceptions` 这类策略必须经过业务和法务确认。
- **可维护性**：不要把所有权限规则写进 Prompt。应沉淀为 policy、attribute、中间件、工具包装器或领域服务。
- **权限边界**：端点级、工具级、数据级、业务级限额要同时存在。尤其 RAG 检索必须考虑文档 ACL，不能先检索全库再让模型“自行判断”。
- **安全风险**：Prompt Injection 防护要靠多层控制：不可信内容隔离、工具白名单、参数校验、审批、合规扫描、审计和评测红队用例。
- **成本控制**：设置模型选择策略。摘要、分类、路由可用更便宜模型；高风险决策或最终生成再使用更强模型。结合 `CompactionProvider` 限制上下文膨胀。
- **可观测性**：审计日志至少包含 userId、tenantId、sessionId、toolName、参数摘要、授权结果、审批人、Purview 结果、token 用量、模型名和耗时。
- **可测试性**：为权限拒绝、越权审批、超限金额、Prompt Injection、敏感信息输出、长上下文压缩建立固定回归用例。
- **凭据治理**：开发可用 `DefaultAzureCredential`，生产应使用明确凭据和最小权限，例如托管身份；API Key 应放入安全配置，不应写入源码或日志。

## 本章小结

- 本章最重要的结论 1：Agent 安全不能只靠 Prompt，必须把身份、scope、业务规则、审批和工具执行边界放在服务端代码中。
- 本章最重要的结论 2：合规治理要覆盖输入和输出两侧，`Microsoft.Agents.AI.Purview` 展示了可包装 ChatClient/Agent 的实现路径。
- 本章最重要的结论 3：成本治理要从上下文压缩、模型分层、调用限额、超时和观测用量共同入手，`CompactionPipeline` 是仓库中最直接的参考样例。
