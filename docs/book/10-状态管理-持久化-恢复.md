# 第 10 章：状态管理、持久化与恢复

> 本章定位：把前面学到的 Session、Workflow 与多 Agent 协作推进到“可中断、可续跑、可恢复”的应用原型阶段。

## 本章目标

学完本章后，你应能够：

- 区分 Agent 会话状态、Provider 状态、Workflow 执行状态、Checkpoint 与 DurableTask 持久化边界。
- 找到仓库中状态读写、序列化、恢复和检查点相关的真实源码入口。
- 跑通或验证持久化会话、工作流 checkpoint/resume、共享状态等 sample。
- 为“企业内部知识与工单协同 Agent 系统”补上任务状态保存、失败恢复和续跑设计。

## 本章与前后章节的关系

- 前置知识：第 7 章的 Session/History/Context/Memory，第 8 章的 Workflow，第 9 章的多 Agent 协作。
- 本章解决：长任务、跨轮次对话、工具调用中断、工作流执行失败后，状态如何保存、如何恢复、哪些内容不会自动恢复。
- 后续衔接：第 11 章会把状态层放入更完整的应用架构；第 16 章会进一步展开 Hosting、DurableTask 与服务化部署。

## 先建立整体认知

在 Agent 应用中，“状态”不是一个单一对象，而是分层存在的：

- **会话状态**：一次 Agent 会话中的历史、Provider 私有状态、Memory 状态等，核心容器是 `AgentSession.StateBag`。
- **聊天历史持久化**：对 ChatClientAgent 来说，历史可以在一轮结束后保存，也可以开启每次服务调用后的中间保存。
- **Workflow 状态**：Workflow 执行过程中，executor 通过 `IWorkflowContext` 读写 scoped state，并在 super step 边界发布更新。
- **Checkpoint**：Workflow 在 super step 结束时导出可恢复的执行状态，用于 resume 或 rehydrate。
- **DurableTask 状态**：面向可部署运行时的持久化编排能力，仓库中已有 `Microsoft.Agents.AI.DurableTask`，但本章只讲入口与边界，深入部署留到第 16 章。

如果缺少状态层，系统会出现这些问题：

- 用户跨轮追问时，Agent 不知道前文。
- 工单流程执行到一半失败后，只能从头再来。
- 工具调用 loop 中断后，无法判断哪些工具已调用、哪些结果已返回。
- 多步骤 Workflow 并行/汇聚时，节点之间无法共享中间结果。

## 核心概念拆解

### 1. `AgentSessionStateBag` 是会话内状态容器

`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentSessionStateBag.cs` 中的 `AgentSessionStateBag` 是线程安全 key-value store。它用字符串 key 存状态值，提供：

- `TryGetValue<T>()`：按类型读取状态。
- `GetValue<T>()`：读取状态，不存在时返回 null。
- `SetValue<T>()`：写入状态。
- `TryRemoveValue()`：删除状态。
- `Serialize()` / `Deserialize()`：转换为 JSON 或从 JSON 恢复。

边界：`StateBag` 只负责存储会话内状态，不等于数据库。真正写文件、写数据库、写 Redis，需要应用层拿到 `SerializeSessionAsync()` 的结果后自行落库。

### 2. `ProviderSessionState<TState>` 是 Provider 私有状态的封装

`dotnet/src/Microsoft.Agents.AI.Abstractions/ProviderSessionState{TState}.cs` 把 Provider 状态管理封装成一个可复用组件。它持有：

- `StateKey`：状态在 `AgentSession.StateBag` 中的 key。
- `GetOrInitializeState()`：不存在时初始化状态并写回 session。
- `SaveState()`：把 Provider 新状态保存回 session。

`dotnet/samples/01-get-started/04_memory/Program.cs` 中的 `UserInfoMemory` 就使用了 `ProviderSessionState<UserInfo>`，说明 Memory 组件状态也可以随 session 一起序列化与恢复。

边界：`ProviderSessionState<TState>` 解决的是“组件如何放入 session”，不是“session 如何持久化到外部系统”。外部持久化仍由调用方负责。

### 3. Agent 会话持久化的入口是 serialize/deserialize

`Agent_Step03_PersistedConversations/Program.cs` 展示了最小链路：

```csharp
AgentSession session = await agent.CreateSessionAsync();
Console.WriteLine(await agent.RunAsync("Tell me a joke about a pirate.", session));
JsonElement serializedSession = await agent.SerializeSessionAsync(session);
AgentSession resumedSession = await agent.DeserializeSessionAsync(serializedSession);
Console.WriteLine(await agent.RunAsync("Now tell the same joke...", resumedSession));
```

这里的关键不是 JSON 打印，而是：应用可以把 `serializedSession` 存入文件或数据库，之后再读出来恢复。

### 4. 工具调用 loop 的中间持久化不是默认语义

`dotnet/samples/02-agents/Agents/Agent_Step19_InFunctionLoopCheckpointing/Program.cs` 使用：

```csharp
RequirePerServiceCallChatHistoryPersistence = true
```

它表示：当 Agent 使用工具时，`FunctionInvokingChatClient` 可能经历“服务调用 -> 工具执行 -> 服务调用”的 loop。开启后，聊天历史会在每次 AI service call 后持久化，而不是等整轮 run 结束后才持久化。

边界很重要：

- 默认更接近“整轮结束后持久化”，语义更原子。
- per-service-call 能提高中断后可观察性和恢复机会。
- 但它也可能保存未完成的中间状态，例如已有 tool call 但 tool result 尚未写入。

### 5. Workflow 状态与 Agent Session 状态不是同一个层次

Workflow 中 executor 通过 `IWorkflowContext` 读写状态。例如 `SharedStates/Program.cs`：

```csharp
await context.QueueStateUpdateAsync(fileID, fileContent, scopeName: FileContentStateConstants.FileContentStateScope, cancellationToken);
var fileContent = await context.ReadStateAsync<string>(message, scopeName: FileContentStateConstants.FileContentStateScope, cancellationToken);
```

这表示 workflow 节点之间共享文件内容，而不是把内容塞进 Agent 的 chat history。Workflow 状态由 `Microsoft.Agents.AI.Workflows.Execution.StateManager` 管理。

### 6. Checkpoint 是 Workflow 恢复边界

`CheckpointAndResume` 与 `CheckpointAndRehydrate` 都强调：checkpoint 在 super step 结束时创建。它保存的是 workflow 可继续执行所需状态。两个恢复路径不同：

- `RestoreCheckpointAsync()`：把 checkpoint 恢复到同一个 `StreamingRun` 实例。
- `InProcessExecution.ResumeStreamingAsync()`：用新的 workflow/run 从 checkpoint 重新水合并继续执行。

边界：checkpoint 不是每一行代码执行后的快照，而是 super step 边界的快照；如果失败发生在一个 super step 内，需要理解可能只能回到上一个 checkpoint。

### 7. DurableTask 是生产级长运行入口，但本章不展开部署

`dotnet/src/Microsoft.Agents.AI.DurableTask` 下有 Durable agent/workflow 支持，例如：

- `DurableAgentSession.cs`：包含 `sessionId` 与 `AgentSessionStateBag`，并提供内部 `Serialize()` / `Deserialize()`。
- `DurableAgentState` 及其 `DurableAgentStateMessage`、`DurableAgentStateResponse` 等：用于表达 durable agent 的请求、响应、消息和内容状态。
- `Workflows/DurableWorkflowClient.cs`、`DurableWorkflowRun.cs`、`DurableWorkflowContext.cs`：面向 durable workflow 的运行入口。

本章只需知道：如果你的目标是跨进程、跨服务实例的长运行恢复，不能只依赖内存 `StreamingRun`；应继续学习第 16 章的 Hosting 与 DurableTask。

## 结合源码理解执行过程

### 1. Agent 会话持久化入口在哪里

入口在 Agent API：

1. `CreateSessionAsync()` 创建 `AgentSession`。
2. `RunAsync(prompt, session)` 让 Agent 在该 session 上运行。
3. `SerializeSessionAsync(session)` 把 session 变为 `JsonElement`。
4. 应用层保存 JSON。
5. `DeserializeSessionAsync(serializedSession)` 恢复 session。
6. 后续 `RunAsync(..., resumedSession)` 继续同一上下文。

`AgentSessionStateBag` 是底层状态容器；具体 Provider 可以把自己的状态通过 `ProviderSessionState<TState>` 写入 `StateBag`。

### 2. Workflow 状态更新如何流转

`StateManager` 的关键路径如下：

1. executor 调用 `context.QueueStateUpdateAsync(...)`。
2. 状态先进入 `StateManager._queuedUpdates`，不是马上成为已发布状态。
3. super step 完成时，运行时调用 `PublishUpdatesAsync()`。
4. `PublishUpdatesAsync()` 按 `ScopeId` 聚合更新，写入 `StateScope`。
5. checkpoint 导出时，`ExportStateAsync()` 要求没有 pending updates；否则会抛出 `Cannot export state while there are queued updates`。
6. 恢复 checkpoint 时，`ImportStateAsync(checkpoint)` 清空当前 scopes 并导入 checkpoint 的 `StateData`。

这个设计说明：Workflow 状态具有“排队更新 -> 步边界发布 -> checkpoint 导出”的阶段性。

### 3. `StatefulExecutor<TState>` 负责什么

`StatefulExecutor<TState>` 是编写有状态 executor 的基类。它提供：

- `ReadStateAsync()`：从 context 读状态，不存在时使用 initial state factory 初始化。
- `QueueStateUpdateAsync()`：把新状态排队写入 context。
- `InvokeWithStateAsync()`：封装“读 -> 改 -> 写”的状态更新模式。
- `StateKey`：默认是 `{ExecutorTypeName}.State`，也可通过 `StatefulExecutorOptions.StateKey` 配置。
- `ScopeName`：通过 `StatefulExecutorOptions` 控制状态 scope。

边界：当 `ConcurrentRunsEnabled` 开启时，executor 不应依赖本地 `_stateCache`；源码中也会在并发运行时避开缓存路径。

### 4. Checkpoint 恢复路径在哪里

`CheckpointAndResume/Program.cs` 的恢复链路：

```csharp
await checkpointedRun.RestoreCheckpointAsync(savedCheckpoint, CancellationToken.None);
await foreach (WorkflowEvent evt in checkpointedRun.WatchStreamAsync())
{
    ...
}
```

`CheckpointAndRehydrate/Program.cs` 的恢复链路：

```csharp
var newWorkflow = WorkflowFactory.BuildWorkflow();
await using StreamingRun newCheckpointedRun =
    await InProcessExecution.ResumeStreamingAsync(newWorkflow, savedCheckpoint, checkpointManager);
```

推荐优先理解 `ResumeStreamingAsync`，因为它更接近“进程重启后重新创建 workflow 再恢复”的真实需求。

## 关键代码或配置讲解

### 代码片段 1：会话序列化

```csharp
JsonElement serializedSession = await agent.SerializeSessionAsync(session);
AgentSession resumedSession = await agent.DeserializeSessionAsync(serializedSession);
```

- 作用：把 `AgentSession` 转为可保存的 JSON，再恢复为 session。
- 为什么重要：这是跨轮次、跨进程恢复 Agent 对话的最小入口。
- 改动它会影响什么：如果不保存 `serializedSession`，程序退出后无法恢复；如果反序列化时使用了不兼容的 Agent/provider，实现可能无法识别旧状态。

### 代码片段 2：Provider 状态读写

```csharp
this._sessionState = new ProviderSessionState<UserInfo>(
    stateInitializer ?? (_ => new UserInfo()),
    stateKey: ...);

var userInfo = this._sessionState.GetOrInitializeState(context.Session);
this._sessionState.SaveState(context.Session, userInfo);
```

- 作用：让 Memory 或自定义 Provider 把私有状态存入 session。
- 为什么重要：复杂 Agent 不只有 chat history，还会有检索偏好、用户信息、工具上下文等状态。
- 改动它会影响什么：`StateKey` 冲突会导致不同 Provider 状态互相覆盖；状态类型不可序列化会影响 session 序列化。

### 代码片段 3：Workflow checkpoint 生成与恢复

```csharp
CheckpointInfo? checkpoint = superStepCompletedEvt.CompletionInfo!.Checkpoint;
await checkpointedRun.RestoreCheckpointAsync(savedCheckpoint, CancellationToken.None);
```

- 作用：在 super step 完成事件中收集 checkpoint，并在需要时恢复。
- 为什么重要：长流程失败后可以从某个稳定边界继续，而不是从头跑。
- 改动它会影响什么：如果没有传入 `CheckpointManager`，就不会生成 checkpoint；如果没有外部保存 checkpoint，进程退出后内存列表也会丢失。

### 代码片段 4：工具 loop 中间持久化

```csharp
RequirePerServiceCallChatHistoryPersistence = true
```

- 作用：每次 AI service call 后都保存聊天历史。
- 为什么重要：多工具调用链中断时，可以看到更细粒度的中间历史。
- 改动它会影响什么：开启后可能保存未完成 tool call 状态；关闭后默认更像整轮 run 结束后统一保存。

## 做一个最小改造实验

### 实验目标

把 `Agent_Step03_PersistedConversations` 从“打印 JSON”改成“写入文件再读回”，模拟真实持久化。

### 改造点

在 sample 中 `SerializeSessionAsync` 后增加：

```csharp
var json = JsonSerializer.Serialize(serializedSession, new JsonSerializerOptions { WriteIndented = true });
await File.WriteAllTextAsync("session.json", json);

var loadedJson = await File.ReadAllTextAsync("session.json");
JsonElement loadedSession = JsonSerializer.Deserialize<JsonElement>(loadedJson);
AgentSession resumedSession = await agent.DeserializeSessionAsync(loadedSession);
```

然后使用 `resumedSession` 继续第二轮 `RunAsync`。

### 你应该观察什么

- 当前目录生成 `session.json`。
- 第二轮回答仍然能引用第一轮上下文。
- 删除 `session.json` 或改坏 JSON 后，恢复会失败。

### 可能出现的问题

- JSON 文件为空或被截断：反序列化抛出 JSON 异常。
- 使用了不同 Agent/provider 恢复旧 session：状态结构不匹配，可能无法恢复完整上下文。
- 状态中包含不可序列化对象：序列化失败或恢复后类型不正确。

## 本章在综合案例中的增量

主案例仍是：**企业内部知识与工单协同 Agent 系统**。

在第 7-9 章之后，系统已经有多轮会话、上下文、一个简单 Workflow 和多 Agent 分工。但它仍然有明显短板：用户提交工单后，如果流程执行到“分类 -> 检索知识 -> 调用工单系统 -> 等待处理”中间失败，系统可能无法知道任务执行到了哪一步。

本章新增能力：**状态层与恢复策略**。

建议引入以下状态对象与边界：

- `TicketSessionState`：保存在 `AgentSession.StateBag` 中，记录用户、当前工单号、最近一次分类、最近一次建议。
- `TicketWorkflowState`：由 Workflow executor 通过 `IWorkflowContext` 管理，记录流程步骤状态，例如 `Classified`、`KnowledgeRetrieved`、`TicketCreated`、`WaitingForApproval`。
- `TicketCheckpointRef`：每个 super step 完成后保存 checkpoint 引用，真实项目中应写入数据库表，例如 `ticket_workflow_checkpoints`。
- `ToolCallAuditState`：对于创建/更新工单这类有副作用工具，记录幂等 key、外部系统返回 id、是否已提交。

加入后，主案例能力变化为：

- 用户关闭页面后，可以用 session JSON 或会话 id 找回上下文。
- 工单 Workflow 失败后，可以从最近 checkpoint 恢复，而不是重新创建重复工单。
- 高风险工具调用中断后，可以根据 audit state 判断“是否已执行外部写操作”。
- 人工审批前的中间状态能被保存，为第 14 章 Human-in-the-Loop 做准备。

当前能力与仓库关系：

- 仓库已有 sample 直接支持：会话序列化、Workflow checkpoint/resume、Workflow shared state。
- 可基于已有 sample 最小改造：把 `serializedSession` 和 `CheckpointInfo` 写入文件或数据库。
- 仓库未直接给出现成业务样例：完整的工单状态表、审批状态机、外部工单系统幂等设计，需要结合后续章节扩展。

## 常见问题与排错

### 问题 1：以为 `SerializeSessionAsync` 会自动写数据库

- 现象：程序退出后无法恢复 session。
- 原因：`SerializeSessionAsync` 只返回 JSON；保存到哪里由应用负责。
- 排查方法：检查是否有 `File.WriteAllTextAsync`、数据库 insert/update、Redis set 等外部持久化代码。
- 修复方式：在每轮关键交互后把 JSON 写入可靠存储，并保存用户 id、会话 id、版本号等索引。

### 问题 2：Workflow 没有产生 checkpoint

- 现象：`checkpoints.Count == 0` 或 `CompletionInfo.Checkpoint` 为 null。
- 原因：运行时没有传入 `CheckpointManager`，或者没有到达 super step 完成事件。
- 排查方法：参考 `CheckpointAndResume`，确认调用 `InProcessExecution.RunStreamingAsync(workflow, input, checkpointManager)`。
- 修复方式：传入 `CheckpointManager.Default` 或自定义 checkpoint manager，并在 `SuperStepCompletedEvent` 中收集 checkpoint。

### 问题 3：恢复 checkpoint 后流程重复执行了有副作用工具

- 现象：外部工单被创建两次，邮件发了两遍，审批单重复提交。
- 原因：checkpoint 是 super step 边界快照，不会天然保证外部系统写操作幂等。
- 排查方法：检查外部工具调用是否有幂等 key、调用结果是否写入状态、恢复点是否位于副作用之前。
- 修复方式：为写操作工具增加业务幂等 key，并在状态中记录外部系统返回 id；恢复后先查状态再决定是否重试。

### 问题 4：读写 Workflow 状态时类型不匹配

- 现象：运行时报 `State for key ... is not of type ...`。
- 原因：同一个 key 在不同 executor 或不同版本代码中使用了不同类型。
- 排查方法：搜索 `QueueStateUpdateAsync`、`ReadStateAsync<T>`、`StateKey`，确认同一 key 的类型一致。
- 修复方式：统一 state key 命名；状态结构变更时做版本迁移，不要直接复用旧 key。

### 问题 5：开启 `RequirePerServiceCallChatHistoryPersistence` 后看到半成品历史

- 现象：历史里出现 tool call，但没有对应 tool result。
- 原因：中断发生在 function loop 中间；该选项允许保存每次服务调用后的中间历史。
- 排查方法：查看 `Agent_Step19_InFunctionLoopCheckpointing` 的注释与输出，确认这是预期语义。
- 修复方式：如果业务需要原子性，关闭该选项，使用默认 end-of-run persistence；如果业务需要可恢复中间态，则增加工具调用状态检查与补偿逻辑。

## 企业级落地提醒

- 稳定性：状态保存要围绕“业务边界”设计，不要只依赖内存对象。长任务至少在每个可重试边界保存状态。
- 可维护性：给状态 key 制定命名规范，例如 `ticket.session.v1`、`ticket.workflow.step.v1`，避免 Provider 或 executor 互相覆盖。
- 权限边界：session JSON 可能包含用户问题、检索内容、工具结果，保存时要加访问控制和租户隔离。
- 安全风险：不要把密钥、临时 token、敏感凭据直接塞进 `StateBag` 或 checkpoint；如果必须保存敏感数据，应加密并设置过期策略。
- 成本控制：过大的 chat history、完整文档内容、冗余工具结果会增加存储与模型上下文成本。应保存必要摘要、外部引用 id，而不是无节制保存全文。
- 可观测性：每次保存 session、生成 checkpoint、恢复 checkpoint、重试工具调用，都应记录日志和 correlation id。
- 可测试性：为“从旧版本状态恢复”“从半完成 tool loop 恢复”“checkpoint 后重试”写集成测试。
- 持久化与恢复：checkpoint 只能恢复框架内可序列化状态；外部系统副作用必须靠业务幂等和补偿机制保证。

## 本章小结

- 本章最重要的结论 1：Agent 会话状态的真实入口是 `AgentSession`、`AgentSessionStateBag`、`ProviderSessionState<TState>` 以及 `SerializeSessionAsync`/`DeserializeSessionAsync`。
- 本章最重要的结论 2：Workflow 状态由 `IWorkflowContext` 与 `StateManager` 管理，checkpoint 在 super step 边界生成，恢复可以走 `RestoreCheckpointAsync` 或 `ResumeStreamingAsync`。
- 本章最重要的结论 3：持久化不是自动等于可靠恢复；外部存储、状态版本、工具幂等、副作用补偿都需要应用层设计。
