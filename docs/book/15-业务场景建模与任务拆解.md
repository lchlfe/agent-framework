# 第 15 章：业务场景建模与任务拆解

> 本章定位：把“会用 Agent 能力”推进到“能把真实业务问题建模成可执行、可验证、可演进的 Agent 系统方案”。

## 本章目标

学完本章后，你应能够：

- 判断一个业务问题是否真的适合用 Agent，而不是普通 API、规则引擎或传统工作流就足够
- 把复杂业务请求拆成意图识别、路由、任务分解、执行、审批、回写、恢复等可落地模块
- 结合仓库现有 sample 与源码，区分哪些任务拆解能力已经直接支持、哪些需要最小改造、哪些属于理论延伸
- 为“企业内部知识与工单协同 Agent 系统”设计出一条可执行、可验证的任务拆解链路

## 先建立整体认知

### 什么叫“业务场景建模”

业务场景建模，不是给场景起个名字，也不是画一个漂亮架构图，而是回答下面几个更实在的问题：

- 用户到底在发起什么类型的请求
- 这个请求里，哪些部分是不确定、开放式、需要模型判断的
- 哪些部分是确定的、应由规则、工作流、数据库或 API 执行的
- 一次请求会经过哪些阶段
- 哪些阶段适合一个 Agent 独立完成，哪些阶段必须拆给多个角色或多个步骤
- 哪些动作必须审批、可重试、可回滚、可审计

如果这几个问题没有被建模清楚，系统通常会出现两种失败：

1. **把本来可以直接编码的事情硬塞给 LLM**，导致成本高、结果不稳定。
2. **把需要 Agent 推理与协作的问题过度规则化**，导致系统脆弱，一遇到复杂输入就失效。

### 什么叫“任务拆解”

任务拆解是把一个用户目标，拆成一组边界清晰、输入输出可定义、可被验证的子任务。对 Agent 系统来说，常见拆解维度包括：

- **按业务阶段拆**：分类 → 检索 → 分析 → 执行 → 审批 → 回写
- **按职责角色拆**：路由 Agent、知识 Agent、工单 Agent、审批 Agent、总结 Agent
- **按风险级别拆**：只读任务自动化，高风险写操作进入审批
- **按执行形态拆**：串行、并发、条件分支、循环迭代、handoff、group chat

### 为什么这一层重要

到了第 15 章，重点已经不是“框架能不能跑”，而是“你能不能把业务做对”。

同样是“处理员工 IT 支持问题”，可能至少有四种完全不同的系统形态：

- 纯 FAQ 问答：一个知识 Agent 即可
- 知识问答 + 工单创建：需要检索 + 工具调用
- 工单分流 + 专家协作：需要 routing + handoff / group chat
- 故障定位 + 高风险变更：需要 workflow + HITL + 状态恢复

如果不做场景建模，你就会把这四类请求混成一个“大而全 Agent”，最后得到一个：

- Prompt 极长
- 工具过多
- 权限混乱
- 输出不可验证
- 失败难排查
- 很难上线的系统

### 在 Agent 系统中它处于哪一层

本章处于“**业务设计层**”和“**编排设计层**”之间：

- 往下，连接 Workflow、Agent、Tool、Memory、审批、状态等具体实现
- 往上，连接业务目标、部门流程、系统边界、风险控制、SLA

它不是一个单独 API，而是一套把框架能力映射到业务问题的方法。

### 如果缺少这一层，会出现什么问题

缺少任务拆解，常见后果有：

- Agent 明明能调用工具，但不知道什么时候调用、调用哪个、先后顺序如何确定
- 有了多个 Agent，却没有清晰职责，结果只是“多个人一起胡说”
- 有了 workflow，却没有业务状态定义，无法恢复与审计
- 能回答知识问题，但一到创建工单、升级工单、通知审批人就断链
- 系统看似跑通 demo，实际接入企业流程后没有权限边界与失败处理

## 核心概念拆解

### 1. 不是所有业务都适合做 Agent

先判断是否适合做 Agent，至少要看四个维度：

#### 适合做 Agent 的问题

- 输入不稳定，用户表达自由度高
- 需要结合知识、上下文、工具、规则共同判断
- 任务路径不是固定死流程，而是存在分流、补充提问、专家协作
- 存在“先理解，再行动”的决策过程

例如：

- 员工描述“电脑最近经常连不上公司 VPN，还提示证书问题”
- 系统需要先判断这是知识问答、故障排查还是应创建工单
- 还要决定是否需要网络专家、终端专家、权限审批人参与

#### 不适合直接做 Agent 的问题

- 输入结构完全确定
- 规则稳定且边界清晰
- 结果要求强一致、低延迟、可完全预测

例如：

- 根据工单 ID 查询状态
- 根据审批单号回写数据库
- 校验某字段格式是否合法

这些更适合直接 API、规则引擎或传统后端服务。

### 2. 业务场景建模的五层拆解法

面对一个真实请求，可以先按下面五层去拆：

#### 第一层：请求类型

用户请求通常可以先被划分为：

- 知识问答
- 工单查询
- 工单创建
- 工单升级 / 转派
- 变更执行申请
- 多部门协同处理

#### 第二层：决策方式

每类请求再判断其决策方式：

- 规则路由：比如根据字段、关键词、来源系统分流
- 结构化输出路由：让模型返回分类结果再走 switch
- handoff：让当前 Agent 把对话移交给更合适的 Agent
- group chat：多个角色围绕同一任务轮流推进

#### 第三层：执行形态

执行形态决定 workflow 长什么样：

- 串行：上一步输出是下一步输入
- 并发：多个只读分析任务可以同时做
- 条件分支：根据分类结果进入不同路径
- 循环：生成 → 审核 → 修正
- 中断恢复：等待审批、人类输入或外部系统回调

#### 第四层：状态对象

任务拆得再漂亮，如果没有状态对象，系统还是不可控。至少要有：

- 会话状态：当前用户、历史消息、上下文
- 任务状态：当前阶段、子任务结果、失败点
- 业务状态：工单号、分类标签、审批结论、检索文档
- 风险状态：是否涉及写操作、是否需要审批、是否允许自动执行

#### 第五层：验证方式

每个子任务必须有可验证结果：

- 分类结果是否结构化
- 检索结果是否有引用来源
- 工具调用是否有参数与返回值
- 审批请求是否能暂停与恢复
- 回写是否能审计

### 3. 常见任务拆解模式

### 串行拆解：Sequential

适合：步骤依赖明显，后一步必须依赖前一步输出。

仓库对应：`AgentWorkflowBuilder.BuildSequential(...)`，源码在：

- `dotnet/src/Microsoft.Agents.AI.Workflows/AgentWorkflowBuilder.cs`

它内部做的事，本质上是：

- 把一组 Agent `BindAsExecutor(...)`
- 用 `WorkflowBuilder.AddEdge(...)` 一个接一个串起来
- 最后把结果交给输出执行器

这类模式适合主案例里的：

- 问题分类 → 检索知识 → 生成答复
- 工单创建 → 路由团队 → 通知用户

### 并发拆解：Concurrent

适合：多个分析任务可以并行进行，然后聚合结果。

仓库对应：`AgentWorkflowBuilder.BuildConcurrent(...)`

源码显示它会：

- 先把输入 fan-out 给多个 Agent
- 再用累加器收集各自输出
- 最后进入一个聚合执行器

这类模式适合主案例里的：

- 同时查询知识库、历史工单、系统状态
- 并发生成“用户可读摘要”和“内部诊断摘要”

### 交接拆解：Handoff

适合：任务主线仍是一段连续对话，但处理者可能需要在多个专家角色之间切换。

仓库对应：

- `dotnet/samples/03-workflows/_StartHere/03_AgentWorkflowPatterns/Program.cs`
- `dotnet/src/Microsoft.Agents.AI.Workflows/HandoffWorkflowBuilder.cs`
- `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/HandoffAgentExecutor.cs`

关键点：

- `HandoffWorkflowBuilder.FunctionPrefix = "handoff_to_"`
- builder 会给可交接目标注册 handoff function
- `HandoffAgentExecutor` 监听这些函数调用
- 一旦检测到 handoff function，被移交目标就接管当前消息链

这类模式适合主案例里的：

- 总入口 Agent 先接单，再交给“知识专家”“网络专家”“权限专家”
- 一个专家回答后，下一轮继续由该专家处理，或者交回 triage Agent

### 群聊拆解：Group Chat

适合：多个角色围绕同一任务轮流推进，不是简单转接，而是共同协商。

仓库对应：

- `GroupChatWorkflowBuilder`
- `GroupChatManager`
- `RoundRobinGroupChatManager`
- `GroupChatToolApproval` sample

源码上，`GroupChatWorkflowBuilder.Build()` 会：

- 创建 host
- 把每个 participant 和 host 双向连接
- 由 `GroupChatManager.SelectNextAgentAsync(...)` 决定谁下一个发言
- 用 `ShouldTerminateAsync(...)` 控制何时结束

这类模式适合主案例里的：

- 网络支持、终端支持、工单协调员围绕复杂故障协同分析
- 在执行高风险动作前，由不同角色补充证据与回滚计划

### 循环拆解：Writer-Critic

适合：不是一次生成就完成，而是要经过质量门禁、反馈修正、再验证。

仓库对应：`07_WriterCriticWorkflow/Program.cs`

这个 sample 值得特别注意，因为它把“任务拆解”从线性步骤升级成了“带质量控制的反馈环”：

- Writer 负责产出
- Critic 负责结构化判断 `Approved / Feedback`
- workflow 使用 `AddSwitch(...)` 决定流向 Summary 或回到 Writer
- `FlowState` 记录迭代次数与历史，避免无限循环

这类模式适合主案例里的：

- 生成工单摘要后由审核 Agent 检查是否缺少关键信息
- 生成知识答复后由策略 Agent 检查是否触碰风险边界

### 条件路由：Routing / Switch

适合：明确存在多个业务分支，而且每个分支需要稳定可解释地落到不同路径。

仓库对应：`ConditionalEdges/02_SwitchCase/Program.cs`

这个 sample 展示了：

- 先让 spam detection agent 输出结构化结果
- 再用 `AddSwitch(...)` 根据 `SpamDecision` 进入不同执行器
- 同一个输入不必走同一流程

对业务建模来说，这就是“显式路由”。

这类模式适合主案例里的：

- 请求是知识问答、普通工单还是高风险变更
- 工单应该转给 Windows、网络、Azure 还是通用支持团队

### 4. Planner 与 Task Decomposition 在本仓库中的真实边界

本章用户要求里提到 planner、task decomposition、routing 等关键词，这里必须明确区分口径。

#### 仓库已有直接支持

- `WorkflowBuilder` / `AgentWorkflowBuilder`：提供任务编排骨架
- `AddSwitch(...)`：提供基于结果的分支路由
- `HandoffWorkflowBuilder`：提供专家交接
- `GroupChatWorkflowBuilder`：提供多角色轮流协作
- `IWorkflowContext` 状态读写：支持任务分解后的共享状态
- `Writer-Critic` sample：展示“带反馈循环的子任务修正”

#### 可基于 sample 改造实现

- 一个专门的 `PlannerAgent`，先产出任务计划，再由 workflow 执行
- 基于 `DeepResearch` sample 的“Manager + Planner + Specialist”模式
- 基于 `CustomerSupport` sample 的“TicketRoutingAgent”做工单分流
- 基于 `SwitchCase` sample 把分类结果映射为不同工具链路

#### 理论延伸，仓库未直接给出完整成品

- 通用自动 planner，把任意业务请求自动分解成任务树并动态执行
- 具备 SLA、成本、权限约束优化的任务调度器
- 能跨多个业务系统自动重写执行计划的自适应编排器

也就是说：**仓库已经给了构建任务拆解系统的大部分积木，但没有一个“万能规划器”现成成品。**

### 5. declarative workflow 如何帮助业务建模

`dotnet/samples/03-workflows/Declarative/CustomerSupport/Program.cs` 很适合本章，因为它已经接近企业业务：

- 有 `SelfServiceAgent`
- 有 `TicketingAgent`
- 有 `TicketRoutingAgent`
- 有 `WindowsSupportAgent`
- 有 `TicketResolutionAgent`
- 有 `TicketEscalationAgent`

这说明什么？

说明任务拆解不一定都要手写 C# Executor 才算合理。很多时候，你先要完成的是：

- 角色定义
- 工具边界定义
- 输出 schema 定义
- 路由规则定义

而 declarative workflow 更适合把这些业务结构表达出来。

同样，`Declarative/DeepResearch/Program.cs` 中的：

- `PlannerAgent`
- `ManagerAgent`
- `KnowledgeAgent`
- `CoderAgent`
- `WeatherAgent`

则非常像“复杂任务分解 + 专家分派”的研究类场景。它并不是直接叫 task decomposition sample，但本质上就是“基于角色的任务拆解”。

## 结合源码理解执行过程

建议按“入口 -> 中间处理 -> 关键对象 -> 输出结果”的顺序理解。

### 1. 入口在哪里

最直接的入口有两类：

#### 入口 A：手写 workflow

比如 `02_AgentsInWorkflows/Program.cs`：

- 构造 `AIAgent`
- 用 `WorkflowBuilder` 建图
- 调用 `InProcessExecution.RunStreamingAsync(...)`
- 用 `TurnToken` 触发执行

#### 入口 B：模式化 builder

比如 `03_AgentWorkflowPatterns/Program.cs`：

- 调用 `AgentWorkflowBuilder.BuildSequential(...)`
- 调用 `AgentWorkflowBuilder.BuildConcurrent(...)`
- 调用 `CreateHandoffBuilderWith(...)`
- 调用 `CreateGroupChatBuilderWith(...)`

这类 builder 相当于“业务拆解模板”。

### 2. 调用是如何继续流转的

以 sequential 为例：

1. `AgentWorkflowBuilder.BuildSequential(...)` 接收一组 Agent
2. 把每个 Agent 通过 `BindAsExecutor(...)` 变成 workflow 可执行节点
3. 用 `WorkflowBuilder.AddEdge(...)` 将节点串起来
4. 最后 `WithOutputFrom(end)` 指定结果输出点
5. 执行时，输入消息进入第一个 Agent
6. 上一个 Agent 的输出成为下一个 Agent 的输入
7. 直到终点输出

以 handoff 为例：

1. `HandoffWorkflowBuilder.WithHandoffs(...)` 注册源 Agent 与目标 Agent 的关系
2. `Build()` 时给每个 Agent 创建 `HandoffAgentExecutor`
3. executor 内部为目标 Agent 生成 handoff function declaration
4. 当前 Agent 运行中如果调用 handoff function
5. `HandoffAgentExecutor` 识别函数名并更新 `HandoffState`
6. workflow 根据 `AddSwitch(...)` 切到下一个目标 executor

以 group chat 为例：

1. `GroupChatWorkflowBuilder.Build()` 创建 host
2. host 与每个 participant 双向连通
3. 每轮由 `GroupChatManager.SelectNextAgentAsync(...)` 选下一位发言者
4. 每次发言后 host 更新历史
5. 若 `ShouldTerminateAsync(...)` 返回 true，则输出结束

### 3. 关键对象分别负责什么

#### `WorkflowBuilder`

负责定义节点、边、分支、输出点，是最底层的通用编排骨架。

#### `AgentWorkflowBuilder`

负责把常见任务拆解模式封装成“现成构造器”，降低搭图成本。

#### `HandoffWorkflowBuilder`

负责把“专家转接”建模为函数调用驱动的动态路由。

#### `GroupChatWorkflowBuilder` + `GroupChatManager`

负责把“多角色讨论”建模成一个可控制迭代次数、可自定义选人逻辑的对话协作系统。

#### `IWorkflowContext`

负责在子任务之间传递状态，是任务拆解真正落地的关键。

`Writer-Critic` sample 里的 `FlowStateHelpers.ReadFlowStateAsync(...)` 和 `SaveFlowStateAsync(...)` 非常适合作为参考。

#### `AddSwitch(...)`

负责把“模型判断结果”安全地落到“流程分支”。这是业务路由从概念走向可执行系统的关键接口。

### 4. 扩展点在哪里

本章最重要的扩展点有四类：

#### 扩展点 A：自定义 Executor

适合把某个业务步骤封装成明确的输入输出节点。

例如：

- `TicketClassificationExecutor`
- `KnowledgeRetrieveExecutor`
- `TicketCreateExecutor`
- `ApprovalGateExecutor`

#### 扩展点 B：自定义 GroupChatManager

适合把“谁该说话”变成业务规则。

`DeploymentGroupChatManager.cs` 就是明确案例：

- 首轮先 QA
- 后续 DevOps

企业场景里可改成：

- 先 triage
- 再知识专家
- 风险命中后转审批专家

#### 扩展点 C：结构化输出 Schema

适合把分类、规划、审批建议、工单摘要等结果变成可验证对象，而不是自由文本。

#### 扩展点 D：Declarative Workflow

适合先把角色与分工建模清楚，再逐步沉淀为代码化实现。

## 关键代码或配置讲解

### 代码片段 1：`AgentWorkflowBuilder.BuildSequential(...)`

- 位置：`dotnet/src/Microsoft.Agents.AI.Workflows/AgentWorkflowBuilder.cs`
- 作用：把一组 Agent 串成顺序执行的 workflow
- 为什么重要：这是“任务按步骤拆解”的最基础实现
- 改动它会影响什么：会影响多 Agent 顺序编排时的消息转发、角色重写与最终输出聚合方式

值得注意的实现细节：

- `AIAgentHostOptions.ReassignOtherAgentsAsUsers = true`
- `AIAgentHostOptions.ForwardIncomingMessages = true`

这意味着在多 Agent 链路里，框架已经帮你处理了“前一个 Agent 的输出如何变成后一个 Agent 可消费输入”的问题。这正是任务拆解能落地的前提之一。

### 代码片段 2：`HandoffWorkflowBuilder.Build()`

- 位置：`dotnet/src/Microsoft.Agents.AI.Workflows/HandoffWorkflowBuilder.cs`
- 作用：把 handoff 关系构造成可执行 workflow
- 为什么重要：很多业务不是严格固定流程，而是“谁更合适谁接手”
- 改动它会影响什么：会影响专家路由逻辑、回到上一个专家的策略、handoff 工具是否对模型可见

这里的核心不是“又多了一个 builder”，而是：**它把专家路由建模成了真实可执行的函数调用协议。**

### 代码片段 3：`HandoffAgentExecutor.HandleAsync(...)`

- 位置：`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/HandoffAgentExecutor.cs`
- 作用：在运行中识别 handoff function，清理内部消息，并切换当前处理 Agent
- 为什么重要：它决定了 handoff 不是概念，而是可运行状态机
- 改动它会影响什么：会影响 handoff 后的消息清洁度、模型是否被内部协议干扰、流式事件是否可见

这里尤其值得注意：

- `HandoffMessagesFilter` 会过滤 handoff 内部函数调用痕迹
- 否则目标 Agent 可能被“内部协议消息”污染，忽略原始用户问题

这对企业系统很重要，因为真实链路里内部控制消息非常多，如果不过滤，效果会明显变差。

### 代码片段 4：`GroupChatWorkflowBuilder.Build()` + `GroupChatManager`

- 位置：
  - `dotnet/src/Microsoft.Agents.AI.Workflows/GroupChatWorkflowBuilder.cs`
  - `dotnet/src/Microsoft.Agents.AI.Workflows/GroupChatManager.cs`
- 作用：把多角色讨论建模成“host + participants + 选人器”
- 为什么重要：不是所有协作都能简化为 handoff，很多复杂任务就是要多角色轮流补充
- 改动它会影响什么：会影响协作轮次、终止条件、上下文保留量与成本

### 代码片段 5：`Writer-CriticWorkflow` 的 `FlowState`

- 位置：`dotnet/samples/03-workflows/_StartHere/07_WriterCriticWorkflow/Program.cs`
- 作用：记录迭代次数与历史
- 为什么重要：任何带反馈环的任务拆解都必须有状态，不然无法控制收敛
- 改动它会影响什么：会影响循环上限、审计能力、恢复能力

### 代码片段 6：`ConditionalEdges/02_SwitchCase` 的 `AddSwitch(...)`

- 位置：`dotnet/samples/03-workflows/ConditionalEdges/02_SwitchCase/Program.cs`
- 作用：把结构化分类结果映射成显式分支
- 为什么重要：这就是业务路由器的最小原型
- 改动它会影响什么：会影响误路由率、默认分支策略和异常兜底体验

## 本章在综合案例中的增量

主案例统一为：**企业内部知识与工单协同 Agent 系统**。

### 本章之前，综合案例处于什么状态

到第 14 章为止，这个系统通常已经具备：

- 能回答企业知识问题
- 能调用企业系统查询或创建工单
- 能接入审批节点处理高风险动作
- 有基础 workflow、多 Agent、状态与上下文概念

但它还缺一个关键能力：

**面对一个真实而复杂的企业请求，系统还不知道应该如何稳定地拆成多个角色和多个步骤。**

### 本章加入了哪一个新能力

本章新增的是：**从业务请求到执行链路的系统化建模与任务拆解能力**。

具体落地为：

- 定义请求类型：知识问答、工单创建、工单升级、变更执行
- 定义路由层：规则 + 结构化输出双层路由
- 定义执行模式：顺序、并发、handoff、group chat、审批中断
- 定义状态对象：请求状态、工单状态、审批状态、协作状态
- 定义最小可验证结果：分类结果、检索结果、执行结果、审批结果

### 这个能力修改了哪些模块、流程或边界

在主案例里，建议明确引入以下模块：

- `RequestTriageAgent`：判断请求类型
- `KnowledgeAgent`：负责企业知识检索与回答
- `TicketRoutingAgent`：负责工单归属团队判断
- `TicketExecutionAgent`：负责创建 / 更新工单
- `SpecialistAgents`：如网络、终端、权限专家
- `ApprovalGate`：高风险动作审批
- `CaseSummaryAgent`：输出用户可读总结

关键流程会从“单一 Agent 直接处理一切”，升级成：

1. 用户提交请求
2. Triage 路由请求类型
3. 只读问题走知识链路
4. 需要行动的问题进入工单链路
5. 复杂问题进入专家协作链路
6. 高风险操作进入审批链路
7. 最终统一总结并回写结果

### 加入后，系统能力发生了什么变化

加入本章能力后，系统从“能做几件事”变成“知道什么时候该做哪件事，并且按正确顺序做”。

收益包括：

- 更容易解释系统为什么这样决策
- 更容易测试各个子路径
- 更容易为高风险路径加审批与审计
- 更容易新增新专家、新路由规则、新任务类型

### 如果不加入这一章的能力，综合案例会卡在哪里

如果没有本章的任务拆解能力，主案例会卡在下面这些点：

- 用户一旦提出复合问题，系统就不知道该先查知识还是先建工单
- 多 Agent 只是“都能说”，却没有明确职责边界
- 审批节点插不进去，或者插进去后流程断裂
- 新增一个支持团队或新业务线时，需要重写大量 Prompt 与流程逻辑

### 明确区分三类内容

#### 1. 仓库已有 sample 直接支持的部分

- sequential / concurrent / handoff / group chat 模式
- 条件路由与 switch-case
- writer-critic 迭代修正
- 审批中断与恢复
- declarative 方式定义 support / research 类角色分工

#### 2. 可以基于已有 sample 做最小改造实现的部分

- 把 `CustomerSupport` 的路由思路换成“企业内部知识与工单协同 Agent 系统”
- 把 `GroupChatToolApproval` 的 QA/DevOps 换成“知识专家 / 工单协调员 / 权限审批人”
- 把 `SwitchCase` 的 spam 分类换成“知识问答 / 工单 / 变更申请”分类
- 把 `Writer-Critic` 的内容修订换成“工单摘要补全 / 风险说明补全”

#### 3. 理论延伸

- 动态 planner 自动拆分任意复杂企业请求
- 根据成本、SLA、权限自动选择执行路径
- 基于历史成功率自动优化任务拆解图

把现有 sample 改造成“企业内部知识与工单协同 Agent 系统”的最小 triage 原型，验证任务拆解不是空谈。

### 改造点

推荐基于：

- `dotnet/samples/03-workflows/ConditionalEdges/02_SwitchCase/Program.cs`

将原有 `SpamDecision` 三分类改成：

- `KnowledgeAnswerExecutor`
- `TicketCreateExecutor`
- `ApprovalRequiredExecutor`

### 操作建议

1. 保留 `AddSwitch(...)` 骨架
2. 把分类 Agent 的 Prompt 改成企业 IT 支持请求分类
3. 定义新的结构化输出对象，例如：
   - `RequestType`
   - `Reason`
   - `RiskLevel`

## 常见问题与排错

### 问题 1：把“多 Agent”误认为“任务拆解已经完成”

- 现象：系统里有多个 Agent，但职责重叠严重，谁都能回答、谁都能调用工具
- 原因：只做了角色命名，没有做输入输出边界设计
- 排查方法：逐个列出每个 Agent 的职责、可见上下文、可调用工具、退出条件
- 修复方式：优先按业务阶段拆，再按角色映射；不要先堆角色再找用途

### 问题 2：把 planner 理解成“必须有一个万能规划 Agent”

- 现象：一上来就想做自动任务树生成器，结果系统非常不稳定
- 原因：跳过了固定模式与半结构化路由阶段
- 排查方法：看当前场景是否真的需要开放式自动规划，还是 switch + handoff 就足够
- 修复方式：先从显式流程、显式 schema、显式分支开始，planner 作为后续增强而不是起点

### 问题 3：路由靠自由文本，导致不可测

- 现象：模型有时说“像是工单”，有时说“建议联系支持”，流程无法稳定触发
- 原因：没有使用结构化输出承载路由决策
- 排查方法：检查分类 Agent 是否有 JSON schema 或明确枚举结果
- 修复方式：使用结构化输出，再通过 `AddSwitch(...)` 显式映射路径

### 问题 4：handoff 后上下文混乱

- 现象：专家 Agent 接手后忽略原问题，开始解释内部转接逻辑
- 原因：handoff 内部函数调用痕迹污染了消息历史
- 排查方法：阅读 `HandoffAgentExecutor.cs` 中 `HandoffMessagesFilter` 相关逻辑，确认是否理解其作用
- 修复方式：保留或仿照框架已有的过滤机制，不要把内部控制消息原样暴露给目标 Agent

### 问题 5：group chat 很热闹，但没有收敛

- 现象：多个 Agent 轮流发言，但没有明确完成条件，成本快速上升
- 原因：没有定义 `ShouldTerminateAsync(...)` 或最大轮次
- 排查方法：检查 `GroupChatManager.MaximumIterationCount`、自定义终止条件、每个角色的发言目标
- 修复方式：先定义“什么算完成”，再允许群聊；没有完成判据就不要用 group chat

### 问题 6：审批被加进去了，但流程恢复不了

- 现象：审批请求发出后，用户批准了，系统却不能从原位置继续
- 原因：没有把审批看成 workflow 中断点，而只是普通对话分支
- 排查方法：参考 `GroupChatToolApproval` sample 中 `RequestInfoEvent` 与 `SendResponseAsync(...)` 的处理方式
- 修复方式：把审批建模为显式中断 / 恢复节点，并记录审批前的任务状态

## 企业级落地提醒

### 1. 稳定性：先固定任务图，再追求自动规划

企业项目早期最容易犯的错，就是追求“什么都自动规划”。

更稳妥的路线是：

- 先把 70% 高频请求做成固定路由图
- 再把剩余模糊问题交给 handoff 或人工复核
- 最后再逐步引入 planner 能力

### 2. 可维护性：任务拆解的边界必须写成文档或 schema

建议至少沉淀：

- 每类请求的入口定义
- 每个 Agent 的职责、工具权限、输入输出 schema
- 每条 workflow 的成功条件、失败条件、恢复点

否则系统规模一大，没人知道为什么请求被分到这条链。

### 3. 权限边界：分类 Agent 不等于执行 Agent

企业里尤其要避免：

- 同一个 Agent 既负责判断风险，又负责执行高风险动作

更推荐：

- `TriageAgent` 负责分类
- `ExecutionAgent` 负责具体执行
- `ApprovalGate` 负责风险闸门

这样更容易审计，也更容易做最小权限控制。

### 4. 安全风险：高风险动作必须脱离自由文本直连

像下面这些动作不要让模型一句话直接执行：

- 开通权限
- 删除数据
- 批量修改工单
- 推送生产变更

正确做法是：

- 先结构化提取意图与参数
- 再走审批 / 策略校验
- 最后由受控工具执行

### 5. 成本控制：group chat 与 planner 都很贵

group chat、研究式 planner、循环修订都很容易放大 token 成本。建议：

- 优先用于高价值、低频、复杂任务
- 对普通高频请求尽量使用分类 + 固定路径
- 限制最大轮次、裁剪历史、按任务级别选择模式

### 6. 可测试性：每个子任务都要能单测与回放

不要只做端到端测试。至少要能单独验证：

- 分类是否稳定
- 路由是否正确
- 审批是否暂停与恢复
- 工具调用参数是否符合预期
- 最终回写是否幂等

## 本章小结

- 本章最重要的结论 1：业务场景建模的核心，不是“做几个 Agent”，而是把请求类型、决策方式、执行形态、状态对象和验证方式定义清楚。
- 本章最重要的结论 2：仓库已经直接支持 sequential、concurrent、handoff、group chat、switch routing、writer-critic loop 等任务拆解基本模式。
- 本章最重要的结论 3：planner 与自动任务分解在仓库里更多体现为可组合能力与参考样例，而不是一个现成万能模块；真正落地应先从显式路由和可验证子任务做起。

下一章会继续把这些已经建模清楚、拆解清楚的能力放入 Hosting、DurableTask 与服务化部署体系，解决“如何长期运行、恢复、上线”的问题。
