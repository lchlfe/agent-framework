# 第 18 章：评测体系与质量保障

> 本章定位：把前面已经能部署、能观测的 Agent 系统，进一步升级为可持续评测、可回归、可发布的生产级系统。

## 本章目标

学完本章后，你应能够：

- 区分 Agent 质量保障中的单元测试、集成测试、conformance 测试、sample 验证与业务评测。
- 在仓库中定位与质量保障直接相关的真实入口：`dotnet/tests`、`dotnet/eng/verify-samples`、`dotnet/samples`。
- 使用 `verify-samples` 对 sample 做自动化验证，理解确定性输出与非确定性 LLM 输出的不同验证方式。
- 为“企业内部知识与工单协同 Agent 系统”设计一套覆盖正确率、稳定性、回归质量与业务效果的评测闭环。
- 判断哪些能力是仓库已有实现，哪些是需要在业务系统中基于框架能力继续扩展的评测实践。

## 本章与前后章节的关系

- 前置知识：第 16 章已经讨论 Hosting、DurableTask 与服务化部署，第 17 章已经讨论日志、追踪与故障排查；没有这些基础，很难定位评测失败原因。
- 本章解决：如何把“能运行”升级为“能被持续验证”，包括 sample 验证、测试分层、回归策略、业务评测集与发布门禁。
- 后续衔接：第 19 章会继续讨论安全、权限、合规和成本治理；其中很多治理策略也应进入评测集和发布检查项。

## 先建立整体认知

Agent 系统的质量保障不能只依赖“我试了一下感觉回答不错”。原因很简单：

- LLM 输出存在非确定性，同一个问题可能有多种合法答案。
- Agent 可能调用工具、检索知识、进入工作流、请求人工审批，失败点比普通 API 更多。
- Prompt、模型版本、Tool schema、RAG 语料、Workflow 节点、持久化策略的微小变化，都可能造成回归。
- 生产级系统除了“答对”，还要关注稳定性、延迟、成本、权限边界和可恢复性。

因此，本章把质量保障拆成四层：

1. **代码级测试**：验证类型、接口、序列化、会话、工具、工作流、checkpoint 等框架行为是否正确。
2. **提供商一致性测试**：验证不同 Agent/provider 的 `RunAsync`、Streaming、Structured Output、Session 等抽象是否满足统一契约。
3. **sample 自动验证**：批量运行 `dotnet/samples`，用确定性断言或 AI 语义评审判断 sample 是否仍然可用。
4. **业务级评测**：围绕真实业务任务建立 golden set、回归集、人工标注、线上反馈和发布门禁。

本仓库已经提供了前三层的重要参考，第四层需要你在业务系统中扩展实现。

  - `dotnet/samples/01-get-started/05_first_workflow`：确定性 workflow sample，可用于最小验证。
  - `dotnet/samples/01-get-started/01_hello_agent`、`02_add_tools`、`03_multi_turn`、`04_memory`：需要模型环境变量的非确定性 Agent sample。

  - `SampleDefinition`：描述 sample 名称、路径、环境变量、输入、确定性断言与语义期望。

  - `VerificationResult`、`MarkdownResultWriter`、`CsvResultWriter`、`LogFileWriter`：沉淀验证结果。

  - `AgentConformance.IntegrationTests.AgentTests<TAgentFixture>`：conformance 测试基类。

  - 在 `GetStartedSamples.All`、`AgentsSamples.All`、`WorkflowSamples.All` 中新增 `SampleDefinition`。

  - 用 `ExpectedOutputDescription` 覆盖 LLM 非确定性语义结果。
  - 用 `RequiredEnvironmentVariables`、`OptionalEnvironmentVariables`、`SkipReason` 控制自动化运行边界。

需要注意：仓库里没有看到独立命名为 `eval` 或 `benchmark` 的完整评测框架；当前最接近“评测体系”的真实入口是 `dotnet/eng/verify-samples` 与 `dotnet/tests`。业务指标、线上 A/B、人工标注平台、成本基准等属于需要基于现有能力扩展的内容。

## 先跑通或先验证一个最小示例

### 示例目标

先用仓库已有工具验证一个确定性 sample，避免一开始就依赖 Azure OpenAI、Foundry、Anthropic 等外部模型环境。

推荐选择：

- sample：`dotnet/samples/01-get-started/05_first_workflow`
- 验证定义：`dotnet/eng/verify-samples/GetStartedSamples.cs`
- 验证工具：`dotnet/eng/verify-samples`

### 运行前提

- 已安装仓库 `global.json` 所要求的 .NET SDK。
- 当前工作目录切到 `dotnet/eng/verify-samples`。
- 确定性 sample 不要求 `AZURE_OPENAI_ENDPOINT`。
- 如果运行 LLM sample，则需要对应环境变量，例如 `AZURE_OPENAI_ENDPOINT`、`AZURE_OPENAI_DEPLOYMENT_NAME`。

### 操作步骤

```bash
cd dotnet/eng/verify-samples

dotnet run -- 05_first_workflow --log results.log --md results.md --csv results.csv
```

如果想验证整个 get-started 分类：

```bash
dotnet run -- --category 01-get-started --parallel 4 --log results.log --md results.md --csv results.csv
```

如果没有设置模型环境变量，工具会跳过需要外部模型的 sample，并在结果中写明缺失变量。

### 预期现象

- `05_first_workflow` 会运行 `dotnet run --framework net10.0`。
- `SampleRunner` 捕获 stdout、stderr、退出码和耗时。
- `SampleVerifier` 检查输出中是否包含：
  - `UppercaseExecutor: HELLO, WORLD!`
  - `ReverseTextExecutor: !DLROW ,OLLEH`
- 控制台显示 `PASSED`，并生成可选的 `results.log`、`results.md`、`results.csv`。

### 成功标志

- 进程退出码为 0。
- Markdown 汇总中该 sample 状态为 passed。
- 没有出现 timeout、缺失必需输出、非零退出码等失败信息。

## 核心概念拆解

### 1. 测试不是评测的全部

`dotnet/tests` 更偏代码级质量保障，适合回答：

- 序列化是否正确？
- structured output 是否能被反序列化？
- session history 是否被正确保存？
- checkpoint resume 是否能恢复 pending request？
- 不同 provider 的 Agent 是否满足统一抽象契约？

例如 `Microsoft.Agents.AI.UnitTests/ChatClient/ChatClientAgent_StructuredOutput_WithRunAsyncTests.cs` 会用 mock `IChatClient` 验证 `RunAsync<T>` 是否设置 JSON Schema response format，并把响应反序列化为目标类型。这类测试确定性强，适合进入每次 PR 的基础门禁。

但 Agent 业务质量还包括“回答是否真正解决业务问题”。这类问题不能完全靠普通单元测试，需要评测集、语义判断、人工复核和线上反馈。

### 2. sample 验证是仓库里的准评测工具

`dotnet/eng/verify-samples` 的设计非常贴近 Agent 评测：

- 对确定性 sample：用 `MustContain`、`MustNotContain` 做精确字符串检查。
- 对非确定性 LLM sample：用 `ExpectedOutputDescription` 描述期望，再由 `SampleVerifier` 中的 `OutputVerifier` Agent 做语义判断。
- 对交互式 sample：通过 `Inputs` 和 `InputDelayMs` 自动输入，例如审批场景传入 `Y`。
- 对不可自动运行 sample：通过 `SkipReason` 明确跳过原因，例如需要外部服务器、Docker、Application Insights、MCP stdio server 等。
- 对环境依赖：通过 `RequiredEnvironmentVariables` 和 `OptionalEnvironmentVariables` 在运行前判断，避免 CI 卡死。

这套工具不是完整的业务评测平台，但它展示了非常关键的模式：**把“怎么判断一个 Agent/sample 是好的”显式写进代码，而不是依赖人工临时观察。**

### 3. conformance 测试保障抽象一致性

`dotnet/tests/AgentConformance.IntegrationTests` 用抽象基类表达 Agent 行为契约。例如 `RunTests<TAgentFixture>` 覆盖：

- `RunWithNoMessageDoesNotFailAsync`
- `RunWithStringReturnsExpectedResultAsync`
- `RunWithChatMessageReturnsExpectedResultAsync`
- `RunWithChatMessagesReturnsExpectedResultAsync`
- `SessionMaintainsHistoryAsync`

这些测试不只是验证某一个 provider，而是验证不同 Agent 实现是否遵守同一套抽象语义。对企业系统来说，如果你支持 Azure OpenAI、Foundry、OpenAI、Anthropic 或内部模型适配器，类似 conformance 测试能防止“换一个模型适配器就破坏上层业务逻辑”。

### 4. 回归质量关注“曾经修过的问题不再回来”

`Microsoft.Agents.AI.Workflows.UnitTests/CheckpointResumeTests.cs` 是典型回归测试。文件注释明确写着它是针对 `GH-2485` 的 regression tests，验证从 checkpoint 恢复后 pending `RequestInfoEvent` 必须重新发出，并且 `RunStatus` 应进入 `PendingRequests`。

这类测试给业务系统的启发是：

- 每次线上事故或严重缺陷都应沉淀成回归用例。
- 回归用例不一定只断言最终答案，也可以断言事件、状态、checkpoint、审批请求、工具调用次数等中间行为。
- 对 Agent 系统来说，中间链路正确往往比最终自然语言文本更稳定、更适合自动化验证。

## 结合源码理解执行过程

建议按“入口 -> 中间处理 -> 关键对象 -> 输出结果”的顺序理解 `verify-samples`。

### 1. 入口在哪里

入口在 `dotnet/eng/verify-samples/Program.cs`。

它做了几件事：

1. 调用 `VerifyOptions.Parse(args)` 解析命令行参数。
2. 定位 `dotnet` 根目录，确认存在 `agent-framework-dotnet.slnx`。
3. 如果存在 `AZURE_OPENAI_ENDPOINT`，构造 `AzureOpenAIClient` 并取得 `ChatClient`。
4. 创建 `SampleVerifier`、`ConsoleReporter`、`VerificationOrchestrator`。
5. 调用 `orchestrator.RunAllAsync(options.Samples, options.MaxParallelism)`。
6. 输出 console summary，并按参数写入 log、CSV、Markdown。
7. 如果有失败结果，返回退出码 1；否则返回 0。

### 2. 调用是如何继续流转的

核心链路如下：

```text
Program.cs
  -> VerifyOptions.Parse(args)
      -> 选择 GetStartedSamples / AgentsSamples / WorkflowSamples
  -> VerificationOrchestrator.RunAllAsync(samples, maxParallelism)
      -> 检查 SkipReason
      -> 检查 RequiredEnvironmentVariables / OptionalEnvironmentVariables
      -> SemaphoreSlim 控制并发
      -> RunSingleAsync(sample)
          -> SampleRunner.RunAsync(projectPath, timeout, inputs)
              -> dotnet run --framework net10.0
              -> 捕获 stdout / stderr / exit code / elapsed
          -> SampleVerifier.VerifyAsync(sample, run)
              -> 检查 exit code
              -> 检查 MustContain
              -> 检查 MustNotContain
              -> 对非确定性输出调用 OutputVerifier Agent
          -> 写入 VerificationResult
  -> ConsoleReporter / LogFileWriter / CsvResultWriter / MarkdownResultWriter
```

这条链路体现了完整的自动化质量保障流程：选择样本、准备运行条件、执行、采集、判定、报告、返回可用于 CI 的退出码。

### 3. 关键对象分别负责什么

- `VerifyOptions`：把命令行变成可执行计划。它支持 `--category`、`--parallel`、`--log`、`--csv`、`--md` 和按 sample 名称过滤。
- `SampleDefinition`：把评测用例数据化。它不是执行器，而是声明“这个 sample 在哪里、需要什么环境、输入什么、期望什么”。
- `SampleRunner`：只负责运行进程，不负责判断正确与否。
- `SampleVerifier`：只负责判断，不负责并发调度。
- `VerificationOrchestrator`：负责编排，包括跳过、并发、超时、收集结果。
- `VerificationResult`：保存最终判定、失败原因、AI reasoning、stdout/stderr 和日志行。
- `MarkdownResultWriter` / `CsvResultWriter`：把结果转成便于人读或流水线消费的产物。

这种分层很值得业务系统借鉴：评测用例、运行器、评审器、报告器应该尽量分开，否则后期很难扩展人工评审、LLM-as-judge、多模型对比或成本分析。

### 4. 扩展点在哪里

如果你新增了 sample 或业务验证场景，最直接的扩展点是新增 `SampleDefinition`。例如：

- 确定性 workflow：设置 `IsDeterministic = true`，用 `MustContain` 覆盖关键输出。
- 非确定性 Agent：设置 `ExpectedOutputDescription`，让 AI verifier 做语义判断。
- 工具调用场景：用 `MustContain` 检查工具输出的固定事实，例如天气工具返回 `cloudy` 和 `15°C`。
- 审批场景：设置 `Inputs = ["Y", "Y"]` 与合适的 `InputDelayMs`。
- 外部依赖场景：设置 `RequiredEnvironmentVariables`，必要时用 `SkipReason` 明确不可自动运行原因。

## 关键代码或配置讲解

### 代码片段 1：`SampleDefinition`

作用：用数据结构描述一个 sample 的验证规则。

关键字段包括：

- `Name`：显示名和过滤名。
- `ProjectPath`：相对 `dotnet/` 的 sample 项目目录。
- `RequiredEnvironmentVariables`：缺失则跳过。
- `OptionalEnvironmentVariables`：缺失也可能跳过，用于避免 sample 进入交互式 fallback 后挂住。
- `SkipReason`：结构性跳过原因。
- `MustContain` / `MustNotContain`：确定性断言。
- `IsDeterministic`：表示不需要 AI 语义验证。
- `ExpectedOutputDescription`：非确定性语义期望。
- `Inputs` / `InputDelayMs`：自动 stdin 输入。

为什么重要：它把“评测标准”从人的脑子里移到了代码中。改动它会直接影响 sample 是否被运行、如何判定通过、失败时报告什么。

### 代码片段 2：`SampleVerifier.VerifyAsync`

作用：执行四类检查：

1. 退出码必须为 0。
2. stdout 必须包含 `MustContain` 中的字符串。
3. stdout 不得包含 `MustNotContain` 中的字符串。
4. 对非确定性 sample，如果有 `ExpectedOutputDescription`，调用 verifier agent 判断语义期望是否满足。

为什么重要：LLM 输出不适合只做全文精确匹配，但也不能完全不测。这里采用“硬断言 + 语义断言”的组合，可以兼顾稳定性与灵活性。

改动它会影响所有 sample 的通过标准，因此要谨慎。企业业务中建议对 verifier prompt、评分维度、阈值和模型版本做版本化管理。

### 代码片段 3：`RunTests<TAgentFixture>`

作用：用统一测试定义约束不同 Agent 实现的基本行为，例如字符串输入、消息输入、多消息输入、session history。

为什么重要：上层业务系统通常不希望关心底层 provider 是 OpenAI、Azure OpenAI、Foundry 还是自定义实现。conformance 测试能确保不同实现至少满足共同契约。

改动它会影响 provider 兼容性判断，适合作为适配新模型或新 Agent 实现时的质量门禁。

### 代码片段 4：`CheckpointResumeTests`

作用：验证 Workflow checkpoint 恢复后的 pending request、状态转换和响应处理是否正确。

为什么重要：生产级 Agent 工作流经常包含长任务、人工审批、外部系统回调和失败恢复。如果恢复后事件丢失或状态错误，最终自然语言回答可能还没出现，系统就已经不可恢复。

改动它会影响 Workflow 持久化恢复的回归保障。

## 做一个最小改造实验

### 实验目标

把 sample 验证工具从“跑已有 sample”扩展成“为综合案例增加一条质量评测用例”的思路实验。

### 改造点

> 注意：本章只讲实验方法。按照本次章节写作任务约束，不在仓库中实际修改 `dotnet/eng/verify-samples` 源码。

假设你已经为“企业内部知识与工单协同 Agent 系统”新增了一个 sample：

```text
dotnet/samples/05-end-to-end/InternalKnowledgeTicketAgent
```

你可以在 `AgentsSamples.All` 或新的 sample 集合中增加类似定义：

```csharp
new SampleDefinition
{
    Name = "InternalKnowledgeTicketAgent_ReturnPolicy",
    ProjectPath = "samples/05-end-to-end/InternalKnowledgeTicketAgent",
    RequiredEnvironmentVariables = ["AZURE_OPENAI_ENDPOINT"],
    OptionalEnvironmentVariables = ["AZURE_OPENAI_DEPLOYMENT_NAME"],
    MustContain = ["ticketId", "policySource"],
    ExpectedOutputDescription =
    [
        "The output should classify the user request as a support or policy question.",
        "The answer should cite an internal policy source rather than inventing a policy.",
        "If the issue requires account changes, the output should indicate that a ticket or approval is required.",
        "The output should not contain stack traces, raw secrets, or unauthorized user data."
    ],
}
```

### 你应该观察什么

- 是否因为缺少环境变量被跳过。
- 是否能在超时时间内完成。
- stdout 是否包含稳定字段，例如 `ticketId`、`policySource`。
- AI verifier 是否认为语义期望满足。
- Markdown/CSV 结果是否足以定位失败原因。

### 可能出现的问题

- 期望写得太模糊，AI verifier 判断不稳定。
- 期望写得太死，合法回答也被判失败。
- sample 输出没有结构化字段，导致只能靠语义判断，回归定位困难。
- 业务 sample 需要外部工单系统或知识库，CI 环境无法稳定运行。

建议做法是：业务 sample 输出尽量包含稳定结构化信号，例如 `classification`、`toolName`、`ticketId`、`approvalRequired`、`citations`，自然语言回答只作为其中一个评测维度。

## 本章在综合案例中的增量

本书主案例是：**企业内部知识与工单协同 Agent 系统**。

在第 16 章和第 17 章之后，主案例已经具备服务化部署和基础可观测性。本章新增的是：**评测与回归保障层**。

新增能力包括：

- 建立离线评测集：覆盖知识问答、工单分类、工具调用、审批触发、恢复续跑等任务。
- 建立 sample 级验证：把可运行的关键业务路径做成 sample，并参考 `verify-samples` 加入自动验证规则。
- 建立回归集：把线上失败、错误分类、错误工具调用、遗漏审批、RAG 引用错误沉淀为固定用例。
- 建立发布门禁：PR 阶段运行单元测试与确定性 sample；预发布阶段运行 LLM sample 和业务评测；生产阶段结合观测数据做持续评估。

能力加入后，系统从“上线后靠人工发现问题”变成“上线前有评测门禁、上线后有回归闭环”。如果不加入这一章能力，综合案例会卡在生产运维阶段：每次改 Prompt、改工具、换模型、更新知识库，都无法判断是否破坏了已有业务能力。

与仓库已有源码的关系：

- 仓库直接支持：`dotnet/tests` 的测试体系、`dotnet/eng/verify-samples` 的 sample 验证模式。
- 可基于已有 sample 最小改造：为综合案例新增 sample，并用 `SampleDefinition` 描述期望。
- 仓库未直接给出现成样例：业务级 golden set、人工标注平台、线上 A/B、业务 KPI 面板，需要项目方自行扩展。

## 常见问题与排错

### 问题 1：运行 `verify-samples` 后大量 sample 被跳过

- 现象：控制台显示 `SKIPPED — Missing required...` 或 `Missing optional...`。
- 原因：sample 依赖环境变量，例如 `AZURE_OPENAI_ENDPOINT`、`AZURE_OPENAI_DEPLOYMENT_NAME`、`AZURE_AI_PROJECT_ENDPOINT` 等。
- 排查方法：查看 skip reason；回到对应 `SampleDefinition` 检查 `RequiredEnvironmentVariables` 和 `OptionalEnvironmentVariables`。
- 修复方式：设置对应环境变量；或者先运行确定性 sample，例如 `05_first_workflow`、`Workflow_StartHere_01_Streaming`、checkpoint 类 workflow sample。

### 问题 2：sample 超时

- 现象：stderr 中出现 `TIMEOUT: Sample did not complete within ...`，退出码为 -1。
- 原因：sample 可能启动了 Web Server、MCP stdio server，或等待用户输入；也可能模型响应过慢。
- 排查方法：检查该 sample 是否本应设置 `SkipReason`；检查是否需要 `Inputs`；检查 `InputDelayMs` 是否太短。
- 修复方式：对不适合自动退出的 sample 设置 `SkipReason`；对交互式 sample 提供 `Inputs`；必要时拆出可自动运行的最小路径。

### 问题 3：确定性断言失败

- 现象：失败信息为 `Output missing expected substring` 或 `Output contains unexpected substring`。
- 原因：sample 输出格式变了、功能回归了，或断言字符串过于脆弱。
- 排查方法：查看 `results.log` 中 stdout；手动运行对应 sample；对比最近改动。
- 修复方式：如果功能正确只是文案变化，更新 `MustContain`；如果核心行为确实变了，修复 sample 或框架代码。

### 问题 4：AI 语义验证不稳定

- 现象：同一个 sample 偶尔 pass，偶尔 fail。
- 原因：LLM 输出本身波动，或 `ExpectedOutputDescription` 太主观；verifier agent 的判断也可能有波动。
- 排查方法：查看 `AIReasoning`；收集多次失败样本；区分“被测 Agent 波动”和“评审 Agent 波动”。
- 修复方式：增加稳定的结构化输出和 `MustContain`；把期望拆成更明确的事实；固定模型版本；对关键业务评测加入人工复核或多评审投票。

### 问题 5：单元测试通过，但业务回答变差

- 现象：`dotnet test` 通过，但用户反馈答案不准确。
- 原因：代码契约没坏，但 Prompt、RAG 语料、工具选择策略或模型行为变化影响了业务效果。
- 排查方法：从线上日志中抽取失败 case，加入业务评测集；对比旧版本和新版本输出。
- 修复方式：建立业务 golden set；把关键业务路径纳入 `verify-samples` 或独立评测流水线；对 Prompt 和知识库更新做灰度发布。

## 企业级落地提醒

### 稳定性

- 不要只测最终回答；也要测工具是否被调用、调用参数是否正确、审批是否触发、checkpoint 是否可恢复。
- 对长任务和人工介入场景，应增加恢复类回归测试，参考 `CheckpointResumeTests` 的思路。
- 对不可自动退出的服务型 sample，要拆出可测试的核心逻辑，不要让 CI 长时间挂起。

### 可维护性

- 评测用例要版本化，最好和 Prompt、工具 schema、知识库版本一起记录。
- 不要让所有断言都依赖自然语言全文匹配；优先输出结构化字段供测试消费。
- 对每个跳过项写清楚 `SkipReason`，否则团队会误以为该路径已被验证。

### 权限边界与安全风险

- 评测集中应包含越权请求、Prompt Injection、敏感信息请求、危险工具调用请求。
- 对工单系统、数据库、邮件、审批等有副作用工具，测试环境必须使用 sandbox 或 mock。
- `verify-samples` 中的 `Inputs = ["Y"]` 适合教学审批 sample，但企业环境不能把审批自动同意作为默认回归路径，应同时测试拒绝、超时、权限不足等分支。

### 成本控制

- LLM 评测会消耗 token，尤其是“被测 Agent + verifier Agent”双模型模式。
- CI 可分层运行：PR 只跑单元测试和确定性 sample；夜间任务跑完整 LLM 评测；发布前跑高优先级业务评测集。
- 对高频回归用例可缓存输入、工具结果和知识检索结果，降低不必要的外部调用。

### 可观测性

- 第 17 章的日志和 tracing 应接入评测结果：失败时不仅要知道“答案错了”，还要知道检索到了什么、调用了哪个工具、审批状态如何、模型返回了什么。
- Markdown/CSV 报告适合做流水线产物，但生产级平台还应把结果送到统一质量看板。

### 可测试性

- 新增业务能力时同步新增评测：新增 Tool 就测 Tool 参数与权限边界；新增 RAG 就测引用与拒答；新增 Workflow 就测状态转换与恢复。
- 对非确定性结果使用“硬约束 + 语义评审”组合：硬约束保证关键事实不丢，语义评审覆盖自然语言表达差异。

## 本章小结

- 本章最重要的结论 1：Agent 质量保障必须分层，`dotnet/tests` 负责代码与契约，`dotnet/eng/verify-samples` 展示 sample 级自动评测，业务系统还要扩展 golden set 与发布门禁。
- 本章最重要的结论 2：非确定性 LLM 输出不能只做全文匹配，应结合 `MustContain`、`MustNotContain`、结构化字段和 AI/人工语义评审。
- 本章最重要的结论 3：回归测试要覆盖中间行为和状态恢复，不应只看最终自然语言回答；checkpoint、审批、工具调用、session history 都是重要质量信号。

下一章将讨论安全、权限、合规与成本治理。它和本章紧密相关：安全与成本不是独立文档，而应进入评测集、回归用例和发布门禁。

## 本章建议产出物

- 一次 `verify-samples` 运行记录，例如 `05_first_workflow` 的 log、Markdown、CSV 汇总。
- 一张评测链路图：`SampleDefinition -> SampleRunner -> SampleVerifier -> VerificationResult -> Markdown/CSV`。
- 一个最小改造实验记录：为主案例设计一条 `InternalKnowledgeTicketAgent_ReturnPolicy` 评测用例。
- 一份问题清单：列出哪些 sample 被跳过、哪些依赖外部环境、哪些评测期望不稳定。
- 一份业务评测集草案：覆盖知识问答、工单分类、工具调用、审批触发、恢复续跑、安全拒答、成本上限。
